## 第一章  JVM 内存模型与垃圾回收

### Q1. JVM 运行时内存区域如何划分？各区域作用是什么？`[阿里][字节][美团]`

**一句话核心**：线程私有：程序计数器 + 虚拟机栈 + 本地方法栈；线程共享：堆 + 方法区（元空间）+ 直接内存。

| 内存区域 | 特点 | 注意点 |
|---------|------|-------|
| 程序计数器 | 线程私有，行号指示器 | JVM 中唯一不会 OOM 的区域，执行 native 时为 Undefined |
| 虚拟机栈 | 线程私有，方法调用栈帧 | StackOverflowError（栈深超限）/ OOM（栈扩展失败）|
| 堆 | 线程共享，对象实例 | GC 主战场，分新生代（Eden+S0+S1）和老年代 |
| 方法区/元空间 | 线程共享，类元数据 | JDK7 叫 PermGen，JDK8 改为 Metaspace（本地内存）|
| 直接内存 | 本机内存，NIO 使用 | -XX:MaxDirectMemorySize 控制上限，绕堆拷贝提升 IO |

**深度解析**：
- **程序计数器**：记录当前线程执行到哪条字节码指令，多线程切换时用于恢复执行位置。
- **虚拟机栈**：每个方法调用 = 一个栈帧（局部变量表 + 操作数栈 + 动态链接 + 方法返回地址）压栈。
- **堆**：几乎所有对象实例在此分配（逃逸分析例外）。对象优先在 Eden 分配，Minor GC 后存活对象进 Survivor，年龄达阈值（默认 15）晋升老年代。
- **方法区**：JDK7 PermGen 在 JVM 堆内有固定上限，OOM 频繁；JDK8 改为 Metaspace 使用本地内存，默认无上限，字符串常量池同时移至堆。
- **直接内存**：NIO 的 DirectByteBuffer 分配，减少堆与 native 层的数据拷贝。

> 💡 **追问**：永久代和元空间的区别？JDK7 为什么把字符串常量池移到堆中（PermGen GC 效率低，OOM 频繁）？JVM 内存模型和 Java 内存模型（JMM）的区别？

### Q2. 对象创建的完整过程（五步法）和 TLAB 机制 `[阿里][腾讯][滴滴]`

**五步流程**：

1. **类加载检查**：遇到 new 指令，检查类是否已加载/验证/准备/解析/初始化，未完成则先执行类加载。
2. **分配内存**：两种策略——①指针碰撞（堆规整，Serial/ParNew GC 后）；②空闲列表（堆碎片化，CMS GC 后）。并发安全：CAS+失败重试 或 **TLAB**（每线程在 Eden 区预分配私有内存块，优先在 TLAB 分配，满了再加锁申请新 TLAB）。
3. **初始化零值**：内存全部置零（int=0，boolean=false，Object=null），字段可不赋初值直接使用。
4. **设置对象头**：Mark Word 记录哈希码、GC 分代年龄（4bit，最大 15）、锁状态；类型指针（klass pointer）指向方法区类元数据。
5. **执行 `<init>`**：按代码执行字段赋值和构造函数，对象才真正可用。

> 💡 **追问**：TLAB（Thread-Local Allocation Buffer）本质是每线程在 Eden 独占一小块内存，优先在 TLAB 分配避免多线程竞争指针。对象一定在堆上分配吗？（不一定：逃逸分析+栈上分配+标量替换）

### Q3. G1 和 CMS 垃圾收集器的核心区别 `[阿里][字节][领英]`

| 对比维度 | CMS | G1 |
|---------|-----|-----|
| 内存模型 | 经典分代（新生代+老年代固定分区）| 堆分成大小相等的 Region，每个 Region 动态扮演 Eden/Survivor/Old/Humongous |
| 回收策略 | 只回收老年代（配合 ParNew 回收新生代）| 整堆管理，优先回收"收益最大"的 Region（Garbage First 命名由来）|
| 停顿预测 | 无法精确预测 | -XX:MaxGCPauseMillis 设目标（默认 200ms），Pause Prediction Model 控制回收数 |
| 内存碎片 | 标记-清除，长期产生碎片触发 Serial Old FullGC | 标记-整理（Region 间复制），基本无碎片 |
| FullGC | Concurrent Mode Failure 退化为单线程 Serial Old | JDK10+ 多线程并行 FullGC |
| 适用场景 | 堆 < 6GB，追求最低延迟 | 堆 > 6GB，可预测停顿（JDK9 起默认 GC）|

> 💡 **追问**：ZGC 的染色指针（Colored Pointer）原理？G1 的 RSet（Remembered Set）解决跨 Region 引用记录问题？何时选 G1，何时选 ZGC？

### Q4. 什么是 STW？GC 如何通过可达性分析找垃圾？ `[百度][字节][京东]`

**STW（Stop-The-World）**：GC 时暂停所有应用线程，防止引用关系变化（漏标/错标），直到 GC 阶段完成才恢复。所有线程须到达"安全点（Safepoint）"才能暂停，Safepoint 设置在方法调用、循环跳转等位置。

**GC Roots（可达性分析起点）**，从此处出发无法到达的对象即为垃圾：
1. 虚拟机栈（栈帧局部变量表）中引用的对象
2. 方法区中类静态属性引用的对象
3. 方法区中常量引用的对象（字符串常量池）
4. 本地方法栈中 JNI 引用的对象
5. JVM 内部引用（Class 对象、常驻异常对象）
6. synchronized 同步锁持有的对象

> 💡 **追问**：三色标记算法（黑=已标记/灰=标记中/白=未标记）？漏标（存活对象被错误回收）如何解决？（写屏障+增量更新/SATB：CMS 用增量更新，G1 用 SATB）

### Q5. 线上 CPU 飙升 100% 的 JVM 排查 SOP `[美团][滴滴][大厂SRE]`

**标准六步排查法**（面试必须流利说出）：

```bash
# 1. 找 Java 进程 PID
jps  或  ps aux | grep java

# 2. 找 CPU 最高的线程 TID（十进制）
top -H -p <PID>

# 3. 十进制 → 十六进制
printf "%x\n" <TID>

# 4. 找对应线程栈（搜索 nid=0x<十六进制>）
jstack <PID> > stack.log

# 5. 查 GC 状态（每秒输出）
jstat -gcutil <PID> 1000

# 6. 查堆对象分布 / 导出 heap dump
jmap -histo:live <PID>
jmap -dump:format=b,file=heap.bin <PID>
# 用 MAT 分析：Dominator Tree + Path to GC Roots
```

**常见根因**：
- 老年代满了，频繁 Full GC（代码有内存泄漏）
- 大量大对象（> `-XX:PretenureSizeThreshold`）直接进老年代
- 业务代码死循环（GC 线程正常，业务线程吃满 CPU）
- 堆配置太小，Minor GC 过于频繁

> 💡 **追问**：内存泄漏（对象被 GC Root 持有无法释放）vs 内存溢出（OOM，内存确实不足）。MAT 分析内存泄漏步骤：Dominator Tree → 找大对象 → Path to GC Roots → 找引用链。

### Q6. JVM 类加载机制和双亲委派模型，如何实现热部署？ `[阿里][腾讯]`

**类加载五个阶段**：加载 → 验证 → **准备**（赋零值）→ 解析 → **初始化**（执行 `<clinit>`，赋真实值）

**双亲委派模型**：
```
Bootstrap ClassLoader（rt.jar，C++实现）
    └── Ext ClassLoader（jre/lib/ext）
            └── App ClassLoader（classpath）
                    └── 自定义 ClassLoader
```
收到加载请求先委托父 ClassLoader，父加载不了（ClassNotFoundException）才自己加载。  
**作用**：String 等核心类永远由 Bootstrap 加载，防止恶意替换；避免类重复加载。

**热部署原理**：自定义 ClassLoader，重新加载时创建新 ClassLoader 实例加载新版本类。旧 ClassLoader 无强引用后 GC 回收。OSGi、Tomcat、Spring Boot DevTools 均基于此。

**Tomcat 为何破坏双亲委派**：WebAppClassLoader 先自己加载 WEB-INF/lib 中的类，目的是隔离不同 webapp（防止不同应用的同名类互相污染）。

> 💡 **追问**：SPI（如 JDBC 驱动）为什么需要线程上下文类加载器？（Bootstrap 加载 JDBC 接口，但无法加载用户 jar 中的驱动实现，需通过 contextCL 委托 AppCL 加载）

### Q7. Java 四种引用类型的区别和使用场景 `[腾讯][京东]`

| 引用类型 | 创建示例 | 回收时机 | 典型场景 |
|---------|---------|---------|---------|
| 强引用 Strong | `Object o = new Object()` | 即使 OOM 也不回收 | 最常用，默认引用 |
| 软引用 Soft | `SoftReference<byte[]> sr = new SoftReference<>(data)` | 内存不足时才回收 | 内存敏感缓存（图片/本地缓存）|
| 弱引用 Weak | `WeakReference<T> wr = new WeakReference<>(obj)` | 下次 GC 必回收 | ThreadLocal 的 ThreadLocalMap key |
| 虚引用 Phantom | `PhantomReference<T> pr = new PhantomReference<>(obj, queue)` | GC 后得到通知（get() 永返 null）| DirectByteBuffer 释放堆外内存 |

**ThreadLocal 内存泄漏原因**：key（ThreadLocal）弱引用，GC 后 key=null；但 value 是强引用无法回收。线程池中线程常驻，脏 Entry 永远积累。**解决：用完必须调用 `threadLocal.remove()`**。

### Q8. JVM 常用调优参数和容器环境配置 `[领益制造][运维必考]`

```bash
# 堆大小（设为相同，避免运行时扩容/缩容抖动）
-Xms4g -Xmx4g

# 线程栈（默认 1M，大量线程时可缩小节省内存）
-Xss512k

# 元空间（防止无限扩张）
-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m

# G1 GC
-XX:+UseG1GC -XX:MaxGCPauseMillis=200
-XX:G1HeapRegionSize=4m   # 1~32MB，建议堆/2048

# GC 日志（生产必配）
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/logs/gc.log
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/logs/heap.hprof

# 容器环境（JDK8u191+ / JDK11+，关键！）
-XX:+UseContainerSupport       # 识别容器内存限制，而非宿主机总内存
-XX:MaxRAMPercentage=75.0      # 使用容器内存的 75% 作为堆上限
```

**容器环境为何必须加 UseContainerSupport**：不加则 JVM 读宿主机全部内存（如 128GB），按此配置堆（如 64GB），远超容器限制（如 2GB），导致被 cgroup OOM Kill。JDK8u191+ 才支持，老项目必须注意版本。

### Q9. OOM 有哪几种类型？如何排查？ `[字节][快手]`

| OOM 类型 | 原因 | 排查方法 |
|---------|------|---------|
| Java heap space | 堆耗尽（泄漏或对象太大）| jmap dump + MAT Dominator Tree |
| GC Overhead Limit Exceeded | GC 耗时超 98% 只回收 <2% | 本质堆不足，同堆 OOM 排查 |
| Metaspace | 动态生成类过多（CGLIB/ASM 泄漏）| `-XX:MaxMetaspaceSize` 限制，检查类加载泄漏 |
| StackOverflowError | 死递归，栈深超限 | jstack 找重复栈帧方法名 |
| Direct buffer memory | NIO 直接内存耗尽 | `-XX:MaxDirectMemorySize`，检查 DirectByteBuffer 释放 |
| unable to create new native thread | 线程数超系统限制 | 减小 -Xss，减少线程数，检查线程泄漏 |

**内存泄漏常见场景**：
1. 静态集合类持有对象不释放
2. 未关闭的连接/流/资源（应使用 try-with-resources）
3. ThreadLocal 未 remove（线程池场景）
4. 缓存无过期时间持续增长
5. 内部类持有外部类引用（匿名内部类、Lambda）
6. 监听器/回调注册未注销

### Q10. 逃逸分析、栈上分配和标量替换 `[阿里][拼多多]`

**逃逸分析**（`-XX:+DoEscapeAnalysis`，默认开启）：JVM 分析对象是否只在方法内使用（不逃逸）。逃逸方式：赋给 static 字段 / 作为返回值 / 传入其他线程 / 赋给外部可见的实例字段。

**三种优化**：
- **栈上分配**：不逃逸的对象直接在栈上分配，方法结束自动回收，减少 GC 压力。
- **标量替换**：把对象拆解为基本类型字段直接在栈上存储（连对象头都省了）。如 `Point(x,y)` → `int x` 和 `int y` 两个变量。
- **同步消除**：不逃逸对象的 `synchronized` 锁操作会被消除，节省开销。

> 💡 **追问**：为什么说"几乎所有对象都在堆上分配"而不是"所有对象"？微服务中大量短生命周期 DTO/Response 对象如何受益于逃逸分析？

### Q11. synchronized 和 ReentrantLock 的区别？AQS 原理？ `[阿里][字节][腾讯]`

| 对比 | synchronized | ReentrantLock |
|-----|-------------|---------------|
| 层面 | JVM 关键字（monitorenter/exit 字节码）| API（基于 AQS）|
| 可中断 | 不支持（阻塞后只能等）| `lockInterruptibly()` 支持 |
| 公平性 | 非公平 | 支持公平模式 `new ReentrantLock(true)` |
| 条件变量 | 一个（Object.wait/notify）| 多个 `Condition`（await/signal）|
| 锁超时 | 不支持 | `tryLock(time, unit)` 支持 |
| 锁升级 | 无锁→偏向锁→轻量级锁→重量级锁 | 无，直接 CAS+AQS 队列 |

**AQS（AbstractQueuedSynchronizer）核心**：
- `state`（volatile int）：同步状态，ReentrantLock 中 = 持锁重入次数

- 独占模式：`tryAcquire` 失败 → 加入队列 → `park` 阻塞；`release` → `unpark` 后继节点
- 子类只需实现 `tryAcquire/tryRelease`，不用管队列

### Q12. volatile 原理？能保证原子性吗？DCL 单例 `[百度][字节][美团]`

**volatile 两大语义**：
- **可见性**：写时 CPU 缓存立刻刷主内存；读时强制从主内存读。底层：LOCK# 前缀指令（x86）或 MFENCE 内存屏障。
- **有序性（禁止重排）**：读屏障（LoadLoad + LoadStore），写屏障（StoreStore + StoreLoad）。

**不保证原子性**：`volatile int i; i++` 包含 read→modify→write 三步，多线程仍会丢失更新。需原子性用 `AtomicInteger`（CAS）或 `synchronized`。

**DCL 双重检查锁**（instance 为何必须 volatile）：
```java
public class Singleton {
    private static volatile Singleton instance; // volatile 必须！
    public static Singleton getInstance() {
        if (instance == null) {                   // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {           // 第二次检查（有锁）
                    instance = new Singleton();   // 非原子操作！
                    // 指令：① 分配内存 ② 初始化对象 ③ 赋引用
                    // 无 volatile 时，③ 可能在 ② 之前，其他线程看到半初始化对象
                }
            }
        }
        return instance;
    }
}
```

### Q13. ThreadLocal 的原理和内存泄漏 `[阿里][腾讯][字节]`

**原理**：每个 Thread 内部有一个 `ThreadLocalMap`（自定义 HashMap，开放地址法解决冲突）。key = ThreadLocal 的弱引用，value = 实际值。`set/get` 操作都针对当前线程的 Map。

**内存泄漏原因**：
1. key（ThreadLocal）是弱引用 → GC 后 key = null，Entry 变"脏"
2. value 是强引用 → 无法被 GC 回收
3. 线程池中线程常驻 → 脏 Entry 永久积累

**解决**：用完后必须调用 `threadLocal.remove()`，尤其是线程池环境（线程复用，不 remove 还会导致数据串行）。

> 💡 **追问**：`InheritableThreadLocal`（子线程继承父线程值）？线程池场景值传递用阿里 `TransmittableThreadLocal`（TTL）？

### Q14. 线程池的核心参数和工作原理 `[美团][滴滴][领益制造]`

**ThreadPoolExecutor 7 个参数**：

| 参数 | 含义 |
|-----|------|
| corePoolSize | 核心线程数，常驻（空闲不销毁，除非 allowCoreThreadTimeOut=true）|
| maximumPoolSize | 最大线程数（超出则拒绝）|
| keepAliveTime + unit | 非核心线程空闲存活时间 |
| workQueue | 任务队列（LinkedBlockingQueue/ArrayBlockingQueue/SynchronousQueue）|
| threadFactory | 线程创建工厂（自定义线程名，方便排查）|
| handler | 拒绝策略（AbortPolicy抛异常/CallerRunsPolicy调用者执行/DiscardPolicy丢弃/DiscardOldestPolicy丢最老）|

**执行流程**：提交任务 → ①核心线程未满则新建 → ②核心线程满则入队 → ③队满则新建非核心线程 → ④达到 maximumPoolSize 则执行拒绝策略。

**为何禁用 Executors 工厂**：
- `FixedThreadPool/SingleThreadExecutor`：无界 LinkedBlockingQueue，可能堆积海量任务 OOM
- `CachedThreadPool`：maximumPoolSize = Integer.MAX_VALUE，可能创建大量线程 OOM

### Q15. CAS 原理和 ABA 问题 `[字节][京东]`

**CAS（Compare And Swap）**：CPU 级原子操作（`cmpxchg` 指令），比较内存值与期望值，相等才写入新值，否则自旋重试。

```java
// AtomicInteger.incrementAndGet() 底层
do {
    old = value.get();
    newVal = old + 1;
} while (!value.compareAndSet(old, newVal));
```

**ABA 问题**：值 A→B→A，CAS 检查仍为 A 认为未变化，中间状态被忽略。  
**解决**：`AtomicStampedReference`（带版本号 stamp）或 `AtomicMarkableReference`（带 boolean 标记）。

**CAS 三个缺点**：
1. 自旋空转浪费 CPU（高并发竞争激烈时）
2. 只能保证单个变量原子操作
3. ABA 问题

> 💡 **追问**：LongAdder vs AtomicLong？LongAdder 分 Cell 数组，减少 CAS 竞争，高并发下吞吐更高，但读需汇总各 Cell。

### Q16. CountDownLatch、CyclicBarrier、Semaphore 的区别 `[腾讯][阿里]`

| 工具 | 特点 | 典型场景 |
|-----|------|---------|
| CountDownLatch | 一次性，计数到 0 触发 await() 释放 | 主线程等 N 个子任务完成 |
| CyclicBarrier | 可重用，所有线程到达栅栏后同时放行，可循环 | N 个线程互相等待到同一个点后同时继续 |

### Q17. Java 内存模型（JMM）核心规则 `[字节][百度]`

1. 程序顺序规则：同一线程前面的操作先于后面的
2. 监视器锁规则：unlock happens-before 后续同一锁的 lock

5. 线程终止规则：子线程操作 happens-before join() 返回
6. 线程中断规则：interrupt() happens-before 检测到中断
7. 对象初始化规则：构造函数 happens-before finalize()
8. 传递性规则：A hb B，B hb C → A hb C

### Q21. StampedLock 和读写锁对比 `[阿里]`

**ReentrantReadWriteLock**：读读不互斥，读写/写写互斥。写线程可能饥饿（大量连续读锁导致写锁无法获取）。

**StampedLock（JDK8+）三种模式**：
1. **写锁**：独占，和 RWL 写锁类似
2. **悲观读锁**：共享，和 RWL 读锁类似
3. **乐观读（核心优化）**：不加锁直接读，读完 `validate(stamp)` 检查是否有写入，有则升级为悲观读锁

```java
StampedLock sl = new StampedLock();
long stamp = sl.tryOptimisticRead();     // 乐观读，不加锁
double x = this.x, y = this.y;          // 直接读
if (!sl.validate(stamp)) {              // 验证：期间是否有写操作
    stamp = sl.readLock();              // 升级为悲观读锁
    try { x = this.x; y = this.y; }
    finally { sl.unlockRead(stamp); }
}
```

> 💡 **缺点**：不支持重入，不支持 Condition，API 复杂易用错。适合读多写极少且读操作极快的场景。

### Q22. Fork/Join 框架和工作窃取算法 `[字节][百度]`

**Fork/Join（JDK7+）**：大任务分解（fork）为子任务并行执行，最终汇总（join）。继承 `RecursiveTask`（有返回值）或 `RecursiveAction`（无返回值）。适合 CPU 密集型递归分治（大数组并行求和、归并排序）。

**工作窃取（Work Stealing）**：每个工作线程维护一个双端 Deque。线程从队列头部（LIFO）取自己的任务；空闲时从其他线程尾部（FIFO）窃取，减少竞争。`ForkJoinPool.commonPool()` 是 Java8 并行流和 CompletableFuture 的默认线程池。

> 💡 **追问**：为什么不建议在 Web 容器中随意使用 `parallelStream()`？共享 `commonPool` 可能与其他任务竞争线程，影响响应时间。建议为业务指定专用线程池。

### Q24. Spring AOP 的实现原理 `[阿里][字节]`

**两种代理方式**：
- **JDK 动态代理**：目标类实现了接口时使用，通过反射生成实现相同接口的代理类，拦截 `invoke()` 方法。
- **CGLIB 代理**：目标类无接口，或 `@EnableAspectJAutoProxy(proxyTargetClass=true)` 时使用，继承目标类生成子类，ASM 字节码重写方法。

**AOP 关键概念**：
- Aspect（切面）= Pointcut（切点，匹配哪些方法） + Advice（通知，增强逻辑）
- `@Before / @After / @AfterReturning / @AfterThrowing / @Around`（最强，可控制执行）

**同类内部调用 AOP 不生效**：`this.method()` 调用的是原始对象，不经过代理。解决：注入自身 `ctx.getBean(XxxService.class).method()` 或 `AopContext.currentProxy()`。

> 💡 Spring AOP 运行时代理；AspectJ 编译时/类加载时织入，功能更强，需 ajc 编译器。

### Q47. Redis 缓存三大问题 `[阿里][美团][领益制造]`

**缓存击穿**（热点 key 过期瞬间）：① 互斥锁重建 ② 逻辑过期（异步更新，不设 TTL）

**缓存雪崩**（大批 key 同时过期或 Redis 宕机）：① TTL 加随机抖动 ② Redis 高可用（哨兵/集群）③ 多级缓存（本地 Caffeine + Redis）

### Q48. Redis 主从、哨兵、Cluster `[腾讯][字节]`

**哨兵**：主观下线 → 客观下线（超 quorum 同意）→ Leader 选举 → 故障转移（Slave 升 Master）。

**Redis Cluster**：16384 slot，`CRC16(key) % 16384` 路由。每节点负责一部分 slot，主从各一。不支持跨 slot 的 multi-key（用 `{tag}` 哈希标签强制同一 slot）。

### Q50. Redis 过期策略和内存淘汰 `[百度][字节]`

**过期删除**：惰性删除（访问时检查）+ 定期删除（每100ms随机抽样）。

**淘汰策略**：`noeviction`（满了报错）/ `allkeys-lru`（缓存推荐）/ `allkeys-lfu`（Redis4+）/ `volatile-lru`（只淘汰设了TTL的key）。

### Q51. Redis 热 key 和大 key `[美团][阿里]`

**热 key**：本地缓存（Caffeine）+ 拆分热 key（`hot_key_0~N`）+ 读写分离。

**大 key**：拆分 + gzip 压缩 + `UNLINK` 异步删除（非阻塞，比 `DEL` 好）。

### Q54. Redis 集群脑裂 `[字节][阿里]`

网络分区后 Sentinel 选出新 Master，旧 Master 仍运行，客户端继续写旧 Master，恢复后数据丢失。

缓解：`min-slaves-to-write 1` + `min-slaves-max-lag 10`（Slave 落后>10s 则 Master 拒绝写）。

## 第六章  分布式系统与消息队列

### Q56. CAP 定理和 BASE 理论 `[阿里][字节]`

**CAP**：C（强一致）/ A（高可用）/ P（分区容错）三选二，P 必须保证，所以只有 CP 和 AP。Zookeeper → CP；Eureka → AP。

**BASE**：Basically Available + Soft State + Eventually Consistent。AP 系统的工程指导原则。

### Q57. 分布式 ID 方案 `[美团][阿里]`

| 方案 | 优点 | 缺点 |
|-----|------|------|
| UUID | 本地生成无依赖 | 无序，索引性能差 |
| 数据库号段模式 | 有序，简单 | 依赖 DB |
| 雪花算法 | 趋势递增，高性能，本地生成 | 依赖时钟（时钟回拨问题）|
| Redis INCR | 简单有序 | 依赖 Redis |

雪花 64 bit：`1符号 + 41时间戳ms + 10机器ID + 12序列号`，单机每ms最多 4096 个ID。

### Q58. 分布式事务方案 `[阿里][领益制造]`

- **2PC（XA）**：同步阻塞，协调者单点，强一致
- **TCC**：Try/Confirm/Cancel，侵入强，性能好，适合核心交易
- **本地消息表**：业务DB和消息同一事务写，定时任务扫描发MQ，消费幂等，最终一致
- **Saga**：多步骤长事务，每步有补偿，适合跨服务长流程

### Q59. Kafka 为何高吞吐 `[阿里][字节][领益制造]`

### Q60. Kafka 消息可靠性 `[阿里][字节]`

- **Producer**：`acks=all` + `retries=INT_MAX` + 幂等（`enable.idempotence=true`）
- **Broker**：`replication.factor=3` + `min.insync.replicas=2`
- **Consumer**：关闭自动提交，业务处理完后手动提交 offset
