## 第三部分：Java 基础知识

### 1. HashMap 的底层实现原理（JDK 8），put 流程，为什么容量必须是 2 的幂次？

**面试官期望答案：**

> JDK 8 的 HashMap 由**数组 + 链表 + 红黑树**组成。put 时先计算 hash，定位桶位置，如果桶为空直接放入；如果不为空，遍历链表（长度 ≥ 8 且数组 ≥ 64 时转红黑树），key 相等则覆盖，不相等则尾插法添加。容量必须是 2 的幂次，因为 `(n - 1) & hash` 等价于取模运算，位运算效率更高，且能保证 hash 分布均匀。

**详细解析：**

**底层结构：**

```
transient Node<K,V>[] table;   // 数组（哈希桶）
// 每个桶中：
static class Node<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;  // 链表
}
// 链表长度 >= 8 且数组长度 >= 64 时转为红黑树：
static final class TreeNode<K,V> extends Node<K,V> { ... }
```

**put 流程（核心源码简化版）：**

```java
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

static final int hash(Object key) {
    int h;
    // 高16位异或低16位，减少碰撞
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;
    
    // 1. 数组为空则初始化
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;
    
    // 2. 计算桶位置，桶为空直接放入
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    else {
        Node<K,V> e; K k;
        // 3. key已存在，直接覆盖
        if (p.hash == hash && ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;
        // 4. 红黑树节点
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
        else {
            // 5. 遍历链表
            for (int binCount = 0; ; ++binCount) {
                if ((e = p.next) == null) {
                    p.next = newNode(hash, key, value, null);
                    // 链表长度 >= 8，尝试转红黑树
                    if (binCount >= TREEIFY_THRESHOLD - 1)
                        treeifyBin(tab, hash);
                    break;
                }
                if (e.hash == hash && ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                p = e;
            }
        }
        if (e != null) {
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null)
                e.value = value;
            afterNodeAccess(e);
            return oldValue;
        }
    }
    ++modCount;
    // 6. 超过阈值则扩容
    if (++size > threshold)
        resize();
    afterNodeInsertion(evict);
    return null;
}
```

**为什么容量必须是 2 的幂次？**

```java
// 计算桶位置的代码
i = (n - 1) & hash;
```

当 `n = 2^m` 时，`n - 1` 的二进制为 m 个 1（如 16 - 1 = 15 = 1111），与 hash 做 `&` 运算等价于对 n 取模：

```
hash % n  等价于  (n - 1) & hash   （仅当 n 是 2 的幂次时成立）
```

好处：
1. **位运算比取模运算快**（CPU 指令周期更少）
2. **hash 分布更均匀**：`n-1` 的低位全是 1，hash 的低位信息全部参与运算

HashMap 在 `resize()` 时会保证新容量也是 2 的幂次：

```java
static final int tableSizeFor(int cap) {
    int n = cap - 1;
    n |= n >>> 1;
    n |= n >>> 2;
    n |= n >>> 4;
    n |= n >>> 8;
    n |= n >>> 16;
    return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;
}
```

这段代码的作用是找到**大于等于 cap 的最小 2 的幂次**。原理是通过不断地将高位 1 向低位传播，最终得到一个全 1 的数，再加 1。

### 2. ConcurrentHashMap 的实现原理，与 HashMap 的区别

**面试官期望答案：**

> JDK 8 的 ConcurrentHashMap 和 HashMap 结构类似（数组 + 链表 + 红黑树），但通过 **CAS + synchronized** 保证线程安全。put 操作时，对数组槽位加 synchronized 锁（锁粒度是桶级别），而不是整个数组。与 HashMap 的主要区别：线程安全、不允许 key/value 为 null、并发扩容时使用多线程协助。

**详细解析：**

**核心数据结构：**

```java
public class ConcurrentHashMap<K,V> extends AbstractMap<K,V>
    implements ConcurrentMap<K,V>, Serializable {
    
    transient volatile Node<K,V>[] table;  // volatile 保证可见性
    // ...
}
```

**put 操作核心流程（简化）：**

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    if (key == null || value == null) throw new NullPointerException();
    int hash = spread(key.hashCode());
    int binCount = 0;
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        // 1. 表为空则初始化
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        // 2. 桶为空，CAS写入
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
                break;  // CAS成功则退出
        }
        // 3. 正在扩容，帮助扩容
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        else {
            // 4. 桶不为空，对头节点加 synchronized 锁
            synchronized (f) {
                if (tabAt(tab, i) == f) {
                    if (fh >= 0) {
                        // 链表操作...
                        binCount = 1;
                    } else if (f instanceof TreeBin) {
                        // 红黑树操作...
                        binCount = 2;
                    }
                }
            }
            // 链表转红黑树
            if (binCount >= TREEIFY_THRESHOLD)
                treeifyBin(tab, i);
        }
    }
    addCount(1L, isContended);  // 更新size，检查是否扩容
    return null;
}
```

**与 HashMap 的区别：**

| 对比项 | HashMap | ConcurrentHashMap |
|-------|---------|-------------------|
| 线程安全 | 不安全 | 安全（CAS + synchronized） |
| null key/value | 允许 1 个 null key，多个 null value | 不允许 null |
| 并发扩容 | 单线程 | 多线程协助扩容（transfer） |
| 锁粒度 | 无锁 | 桶级别（synchronized 头节点） |
| size() | 直接返回 size | 使用 CounterCell + CAS 统计 |

**size() 的实现（高并发场景，第 8 部分展开）：**

ConcurrentHashMap 使用 `LongAdder` 思想，维护一个 `baseCount` + `CounterCell[]`。每个线程在自旋 CAS 失败后，会找一个 CounterCell 进行累加，避免竞争。

### 3. ArrayList vs LinkedList 区别及适用场景

**面试官期望答案：**

> ArrayList 底层是**动态数组**，支持随机访问 O(1)，插入/删除中间元素需要移动元素 O(n)。LinkedList 底层是**双向链表**，插入/删除只需修改指针 O(1)（已知位置），随机访问需要遍历 O(n)。大多数场景下 ArrayList 性能更好（缓存友好），LinkedList 适合频繁头尾操作的场景。

**详细解析：**

**ArrayList 核心源码：**

```java
public class ArrayList<E> extends AbstractList<E> {
    transient Object[] elementData;  // 底层数组
    private int size;
    
    // 扩容：1.5倍
    private void grow(int minCapacity) {
        int oldCapacity = elementData.length;
        int newCapacity = oldCapacity + (oldCapacity >> 1);  // 1.5倍
        elementData = Arrays.copyOf(elementData, newCapacity);
    }
}
```

**LinkedList 核心源码：**

```java
public class LinkedList<E> extends AbstractSequentialList<E> {
    transient Node<E> first;
    transient Node<E> last;
    
    private static class Node<E> {
        E item;
        Node<E> next;
        Node<E> prev;
    }
}
```

**性能对比：**

| 操作 | ArrayList | LinkedList |
|-----|-----------|------------|
| get(i) 随机访问 | O(1) | O(n) |
| add(e) 尾部添加 | O(1) 均摊 | O(1) |
| add(i, e) 中间插入 | O(n) | O(n)（查找位置）+ O(1)（插入） |
| remove(i) 中间删除 | O(n) | O(n)（查找）+ O(1)（删除） |
| 内存占用 | 紧凑（连续内存） | 分散（每个节点额外 2 个引用） |

**为什么大多数情况下 ArrayList 更好？**

1. **CPU 缓存友好**：ArrayList 的数据在内存中连续存储，能充分利用 CPU 缓存行（Cache Line），遍历速度远快于 LinkedList
2. **内存开销小**：LinkedList 每个节点需要额外的 prev/next 指针（64 位 JVM 上每个 Node 多 16 字节）
3. **JVM 优化**：Arrays.copyOf 等底层操作经过高度优化

**适用场景：**
- **ArrayList**：绝大多数场景，读多写少，随机访问多
- **LinkedList**：频繁在头尾插入/删除（如实现队列、栈），或需要频繁在已知位置插入/删除

### 4. String、StringBuilder、StringBuffer 的区别

**面试官期望答案：**

> String 不可变（final），每次操作创建新对象。StringBuilder 可变，非线程安全，性能最好。StringBuffer 可变，线程安全（synchronized），性能稍差。拼接字符串优先用 StringBuilder。

**详细解析：**

```java
// String 不可变
public final class String {
    private final char[] value;  // JDK8
    // private final byte[] value;  // JDK9+
}

// StringBuilder 非线程安全
public final class StringBuilder extends AbstractStringBuilder {
    // append() 无 synchronized
}

// StringBuffer 线程安全
public final class StringBuffer extends AbstractStringBuilder {
    @Override
    public synchronized StringBuffer append(String str) { ... }
}
```

**性能对比：**

```java
// 慢！每次循环都创建新的 String 对象
String s = "";
for (int i = 0; i < 10000; i++) {
    s += i;  // 编译后等价于 s = new StringBuilder().append(s).append(i).toString()
}

// 快！只有一个 StringBuilder 对象
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
    sb.append(i);
}
```

> **面试加分点**：提到 JDK 9+ String 底层从 `char[]` 改为 `byte[]` + `coder`（LATIN1/UTF16），减少了内存占用（纯英文字符只需 1 字节/字符而非 2 字节）。

### 5. Java 中的四种引用类型（强、软、弱、虚）及使用场景

**面试官期望答案：**

> **强引用**：`Object obj = new Object()`，只要引用存在就不会被 GC。**软引用**：`SoftReference`，内存不足时才回收，适合做缓存。**弱引用**：`WeakReference`，下次 GC 一定回收，适合做临时缓存（WeakHashMap、ThreadLocal）。**虚引用**：`PhantomReference`，不影响 GC，仅用于跟踪回收（如 Cleaner 释放堆外内存）。

**详细解析：**

| 引用类型 | 回收时机 | 使用场景 | 示例 |
|---------|---------|---------|------|
| 强引用 Strong | 永不回收（除非断开引用） | 日常编码 | `Object obj = new Object()` |
| 软引用 Soft | 内存不足时回收 | 本地缓存、图片缓存 | `SoftReference<byte[]>` |
| 弱引用 Weak | 下次 GC 时回收 | WeakHashMap、ThreadLocal 的 key | `WeakReference<Object>` |
| 虚引用 Phantom | 随时回收，仅通知 | 堆外内存释放（DirectByteBuffer） | `PhantomReference<Object>` |

```java
// 软引用 —— 内存缓存
Object obj = new Object();
SoftReference<Object> softRef = new SoftReference<>(obj);
// 内存不足时 obj 会被回收，softRef.get() 返回 null
obj = null;  // 断开强引用

// 弱引用 —— WeakHashMap 的 key 就是弱引用
WeakReference<Object> weakRef = new WeakReference<>(obj);
// 下次 GC 时 obj 会被回收，weakRef.get() 返回 null

// 虚引用 —— 必须配合 ReferenceQueue
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);
// get() 永远返回 null
// GC 回收后，phantomRef 会被放入 queue，在回调中释放堆外资源
```

### 6. 接口和抽象类的区别

**面试官期望答案：**

> 接口定义行为契约（"能做什么"），抽象类定义共同模板（"是什么"）。接口支持多继承（一个类实现多个接口），抽象类只能单继承。JDK 8 后接口可以有 default 方法。选接口当需要多实现，选抽象类当需要共享代码。

**详细解析：**

| 对比项 | 接口 interface | 抽象类 abstract class |
|-------|--------------|---------------------|
| 继承 | 一个类可实现多个接口 | 一个类只能继承一个抽象类 |
| 构造方法 | 不能有 | 可以有 |
| 成员变量 | 只能是 public static final（常量） | 可以有各种成员变量 |
| 方法 | JDK8 前全抽象，JDK8 后有 default/static | 可以有抽象方法和具体方法 |
| 设计意图 | 定义行为契约（"能做什么"） | 定义模板和共享代码（"是什么"） |

```java
// 接口 —— 定义能力
public interface Flyable {
    void fly();  // 抽象方法
    
    default void describe() {
        System.out.println("我能飞");
    }
}

// 抽象类 —— 定义模板
public abstract class Bird {
    protected String name;
    
    public Bird(String name) {
        this.name = name;
    }
    
    // 具体方法 —— 子类共享
    public void eat() {
        System.out.println(name + " 在吃东西");
    }
    
    // 抽象方法 —— 子类必须实现
    public abstract void sing();
}
```

### 7. equals() 和 == 的区别，hashCode() 和 equals() 的关系

**面试官期望答案：**

> `==` 比较基本类型时比较值，比较引用类型时比较地址。`equals()` 是 Object 的方法，默认比较地址，通常需要重写来比较内容。**两个对象 equals() 相等，则 hashCode() 必须相等**（用于 HashMap 等）。hashCode() 相等不代表 equals() 相等（哈希碰撞）。

**详细解析：**

```java
// == 比较地址
String a = new String("hello");
String b = new String("hello");
System.out.println(a == b);      // false（不同对象）
System.out.println(a.equals(b)); // true（内容相同，String重写了equals）

// 字符串常量池
String c = "hello";
String d = "hello";
System.out.println(c == d);      // true（指向常量池同一个对象）
```

**正确重写 equals() 和 hashCode()：**

```java
public class User {
    private String name;
    private int age;
    
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        User user = (User) o;
        return age == user.age && Objects.equals(name, user.name);
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }
}
```

**为什么要同时重写？**

如果只重写 `equals()` 不重写 `hashCode()`：
```java
Set<User> set = new HashSet<>();
set.add(new User("张三", 25));
set.add(new User("张三", 25));  // equals() 返回 true，但 hashCode() 不同
// 结果：set 中有两个元素！违反了 Set 不重复的约定
```

### 8. Java 中的异常体系结构，checked vs unchecked exception

**面试官期望答案：**

> Java 异常体系顶层是 `Throwable`，分为 `Error`（不可恢复）和 `Exception`。Exception 又分为 `checked`（编译时检查，必须处理或声明）和 `unchecked`（运行时异常，RuntimeException）。常见 checked：IOException、SQLException。常见 unchecked：NullPointerException、IllegalArgumentException。

**详细解析：**

```
Throwable
├── Error（不可恢复，如 OOM、StackOverflow）
│   ├── OutOfMemoryError
│   └── StackOverflowError
└── Exception
    ├── Checked Exception（编译时检查）
    │   ├── IOException
    │   ├── SQLException
    │   └── FileNotFoundException
    └── RuntimeException / Unchecked Exception（运行时）
        ├── NullPointerException
        ├── ClassCastException
        ├── IllegalArgumentException
        ├── IndexOutOfBoundsException
        └── ConcurrentModificationException
```

**checked vs unchecked：**

```java
// checked —— 必须处理（try-catch 或 throws）
public void readFile() throws IOException {  // 必须声明
    FileInputStream fis = new FileInputStream("test.txt");
}

// unchecked —— 不强制处理
public void divide(int a, int b) {
    int c = a / b;  // 可能抛出 ArithmeticException，但不需要声明
}
```

**最佳实践：**
- 用 checked exception 处理**可恢复**的错误（如文件不存在，可以让用户重新选择文件）
- 用 unchecked exception 处理**编程错误**（如传入 null 参数）
- 不要用异常控制业务流程
- 不要 catch 后什么都不做（吞异常）

### 9. Java 泛型的类型擦除及其影响

**面试官期望答案：**

> Java 泛型在编译时进行类型检查，但编译后**擦除为原始类型**（擦除到上限，通常是 Object）。运行时无法获取泛型类型信息（`new T()` 不行）。这导致：不能创建泛型数组、不能 instanceof 泛型类型、运行时类型参数丢失。

**详细解析：**

```java
// 编译前
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// 编译后（类型擦除后）
List list = new ArrayList();       // 擦除为 raw type
list.add("hello");
String s = (String) list.get(0);   // 编译器自动插入强制转换
```

**类型擦除的影响：**

```java
// 1. 不能创建泛型数组
// List<String>[] array = new List<String>[10];  // 编译错误
List<String>[] array = new List[10];  // 可以，但有 unchecked 警告

// 2. 不能 instanceof 泛型
// if (list instanceof List<String>) {}  // 编译错误

// 3. 不能 new T()
public <T> T create() {
    // return new T();  // 编译错误
    // 解决方案：传入 Class<T> 参数
    return null;
}

// 4. 两个不同泛型参数的 List 在运行时是同一个类型
List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
System.out.println(a.getClass() == b.getClass());  // true！都是 ArrayList
```

**获取泛型类型的方式：**

```java
// 通过子类保留泛型信息（TypeToken 模式）
public abstract class TypeToken<T> {
    private final Type type;
    protected TypeToken() {
        Type superclass = getClass().getGenericSuperclass();
        this.type = ((ParameterizedType) superclass).getActualTypeArguments()[0];
    }
    public Type getType() { return type; }
}

// 使用
TypeToken<List<String>> token = new TypeToken<List<String>>() {};
System.out.println(token.getType());  
// java.util.List<java.lang.String>
```

### 10. Java 反射的原理及应用场景，性能优化方案

**面试官期望答案：**

> 反射是在运行时获取类的信息并操作类的属性和方法。通过 Class 对象获取 Field、Method、Constructor，可以绕过访问控制。应用场景：框架（Spring IoC、MyBatis）、注解处理、动态代理。性能优化：缓存 Method/Field 对象、使用 MethodHandle、关闭安全检查（setAccessible）。

**详细解析：**

```java
// 反射的基本使用
Class<?> clazz = Class.forName("com.example.User");

// 创建实例
Object obj = clazz.getDeclaredConstructor().newInstance();

// 调用方法
Method setName = clazz.getDeclaredMethod("setName", String.class);
setName.invoke(obj, "张三");

// 访问私有字段
Field nameField = clazz.getDeclaredField("name");
nameField.setAccessible(true);  // 关闭访问检查
nameField.set(obj, "李四");
```

**常见应用场景：**
1. **Spring IoC**：通过反射创建 Bean、注入依赖
2. **MyBatis**：通过反射将 ResultSet 映射为 Java 对象
3. **JDBC**：`Class.forName("com.mysql.cj.jdbc.Driver")` 加载驱动
4. **注解处理**：运行时读取注解信息
5. **序列化/反序列化**：Jackson、Gson 通过反射读写字段

**性能优化方案：**

```java
// 1. 缓存 Method/Field 对象（最重要！）
private static final Map<String, Method> METHOD_CACHE = new ConcurrentHashMap<>();

public static Object invokeMethod(Object obj, String methodName, Object... args) 
    throws Exception {
    String key = obj.getClass().getName() + "." + methodName;
    Method method = METHOD_CACHE.computeIfAbsent(key, k -> {
        Method m = obj.getClass().getDeclaredMethod(methodName, ...);
        m.setAccessible(true);
        return m;
    });
    return method.invoke(obj, args);
}

// 2. 使用 MethodHandle（JDK 7+，性能接近直接调用）
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle handle = lookup.findVirtual(obj.getClass(), "methodName", 
    MethodType.methodType(returnType, paramTypes));
handle.invoke(obj, args);

// 3. setAccessible(true) 减少安全检查开销
```

> **反射 vs 直接调用的性能差距**：JDK 8 中反射调用大约是直接调用的 10~100 倍慢（无缓存）。使用缓存后差距缩小到 2~5 倍。JDK 9+ 通过 Module 系统对反射做了更多限制。

### 11. Java IO 流分类，NIO 与 BIO 的区别

**面试官期望答案：**

> IO 流按数据单位分字节流和字符流，按方向分输入和输出流。BIO（Blocking IO）是同步阻塞模型，一个连接一个线程。NIO（Non-blocking IO / New IO）是同步非阻塞模型，通过 Selector 实现一个线程管理多个 Channel。NIO 的核心组件：Channel、Buffer、Selector。

**详细解析：**

**IO 流体系：**

```
InputStream（字节输入）
├── FileInputStream
├── BufferedInputStream
├── ObjectInputStream
└── ByteArrayInputStream

OutputStream（字节输出）
├── FileOutputStream
├── BufferedOutputStream
├── ObjectOutputStream
└── ByteArrayOutputStream

Reader（字符输入）
├── FileReader
├── BufferedReader
└── InputStreamReader

Writer（字符输出）
├── FileWriter
├── BufferedWriter
└── OutputStreamWriter
```

**BIO vs NIO：**

```java
// BIO —— 一次读写一个线程
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket socket = server.accept();  // 阻塞等待连接
    new Thread(() -> {
        InputStream is = socket.getInputStream();
        // 读取数据，阻塞等待...
    }).start();
}

// NIO —— 一个线程管理多个 Channel
Selector selector = Selector.open();
ServerSocketChannel ssc = ServerSocketChannel.open();
ssc.configureBlocking(false);  // 非阻塞
ssc.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();  // 阻塞直到有事件
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            // 接受新连接
        } else if (key.isReadable()) {
            // 读取数据
        }
    }
}
```

| 对比项 | BIO | NIO |
|-------|-----|-----|
| 模型 | 同步阻塞 | 同步非阻塞 |
| 线程 | 一个连接一个线程 | 一个线程管理多个连接 |
| 核心 | Stream | Channel + Buffer + Selector |
| 方向 | 单向 | 双向（Channel） |
| 适用 | 连接数少、数据量大 | 连接数多、数据量小 |

### 1. JVM 内存模型（堆、栈、方法区、程序计数器、本地方法栈）各区域的作用

**面试官期望答案：**

> JVM 内存分为：**堆**（对象实例，GC 主要区域）、**方法区/元空间**（类信息、常量、静态变量）、**虚拟机栈**（每个线程的栈帧，存储局部变量表、操作数栈）、**本地方法栈**（Native 方法）、**程序计数器**（当前线程执行的字节码行号）。其中堆和方法区是线程共享的，栈和程序计数器是线程私有的。

**详细解析：**

```
├──────────────────┼──────────────────────────┤
│                  │  程序计数器（PC Register）   │
│                  │  ├─ 记录当前执行的字节码行号  │
│                  ├──────────────────────────┤
│                  │  虚拟机栈（VM Stack）        │
│                  │  ├─ 栈帧（每次方法调用）      │
│   堆（Heap）      │  │  ├─ 局部变量表           │
│   ├─ 新生代       │  │  ├─ 操作数栈             │
│   │  ├─ Eden      │  │  ├─ 动态链接             │
│   │  ├─ S0/S1     │  │  └─ 方法返回地址          │
│   │  └─ ...       │  ├──────────────────────────┤
│   └─ 老年代       │  本地方法栈（Native Stack）   │
│                  │  └─ 为 Native 方法服务       │
├──────────────────┼──────────────────────────┤
│  方法区/元空间     │                          │
│  ├─ 类信息         │                          │
│  ├─ 常量池         │                          │
│  ├─ 静态变量       │                          │
│  └─ JIT 编译代码   │                          │
└──────────────────┴──────────────────────────┘
```

| 区域 | 线程共享 | 存储内容 | 异常 |
|------|---------|---------|------|
| 堆 | 是 | 对象实例、数组 | OOM |
| 方法区/元空间 | 是 | 类信息、常量、静态变量 | OOM |
| 虚拟机栈 | 否 | 栈帧（局部变量、操作数栈） | StackOverflow / OOM |
| 本地方法栈 | 否 | Native 方法信息 | StackOverflow / OOM |
| 程序计数器 | 否 | 字节码行号 | 无 |

> **注意**：JDK 8 开始，方法区的实现从永久代（PermGen）改为了元空间（Metaspace），使用**本地内存**而非 JVM 堆内存，解决了永久代 OOM 和大小调优的问题。

### 2. JVM 类加载机制（双亲委派模型）及其作用

**面试官期望答案：**

> 类加载过程：加载 → 链接（验证、准备、解析）→ 初始化。双亲委派模型：当一个类加载器收到加载请求时，先委派给父加载器，只有父加载器无法加载时才自己加载。加载器层级：Bootstrap → Extension → Application → 自定义。作用：防止核心类被篡改（如自定义 `java.lang.String`）。

**详细解析：**

**类加载过程：**

```
加载 → 链接 → 初始化
        │
        ├─ 验证：验证字节码合法性
        ├─ 准备：为静态变量分配内存并设默认值（0/null）
        ├─ 解析：符号引用 → 直接引用
        └─ 初始化：执行 <clinit>，赋真正的值
```

```java
// 准备阶段 vs 初始化阶段
public class Test {
    // 准备阶段：value = 0（默认值）
    // 初始化阶段：value = 100（赋值）
    public static int value = 100;
    
    // 准备阶段：final 常量直接赋值为 123（编译期常量）
    public static final int CONST = 123;
}
```

**双亲委派模型：**

```
        Bootstrap ClassLoader（C++ 实现，加载 rt.jar）
                ↑ 委派
        Extension ClassLoader（加载 ext 目录下的 jar）
                ↑ 委派
        Application ClassLoader（加载 classpath 下的类）
                ↑ 委派
        Custom ClassLoader（自定义）
```

```java
// 双亲委派的核心代码（ClassLoader.loadClass）
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 委派给父加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) { }
            
            // 3. 父加载器无法加载，自己加载
            if (c == null) {
                c = findClass(name);
            }
        }
        return c;
    }
}
```

**打破双亲委派的场景：**
1. SPI（如 JDBC、JNDI）—— 线程上下文类加载器
2. Tomcat —— webapp 类加载器优先加载应用类
3. OSGi —— 模块化类加载
4. 热部署

### 3. 垃圾回收算法及垃圾收集器

**面试官期望答案：**

> 垃圾回收算法：标记-清除（有碎片）、标记-整理（无碎片但移动对象）、复制算法（Eden/Survivor）。垃圾收集器：CMS（低延迟，标记-清除）、G1（分区域、可预测停顿）、ZGC（超低延迟，<10ms）。G1 是 JDK 9+ 默认收集器。

**详细解析：**

**三种基本算法：**

```
标记-清除：
[已用][已用][  碎片  ][已用][  碎片  ]
  ↑         ↑         ↑
标记存活    清除死亡    产生内存碎片

标记-整理：
[已用][已用][已用][         空闲          ]
存活对象向一端移动，无碎片，但移动成本高

复制算法：
将内存分为两块，每次只使用一块，GC 时将存活对象复制到另一块
适用于新生代（Eden + 2 个 Survivor）
```

**垃圾收集器演进：**

| 收集器 | 分代 | 算法 | 特点 | 适用 |
|-------|------|------|------|------|
| Serial | 新生代 | 复制 | 单线程，STW | 客户端模式 |
| Parallel Scavenge | 新生代 | 复制 | 多线程，吞吐量优先 | 后台计算 |
| CMS | 老年代 | 标记-清除 | 低延迟，4 个阶段 | JDK 8 常用 |
| G1 | 全堆 | 复制+标记-整理 | 分区域，可预测停顿 | JDK 9+ 默认 |
| ZGC | 全堆 | 着色指针+读屏障 | 超低延迟 <10ms | JDK 15+ |

**G1 收集器详解（重点）：**

```
G1 将堆划分为多个 Region（通常 1~32MB，2048 个）
每个 Region 可以是 Eden、Survivor、Old 或 Humongous
G1 跟踪每个 Region 的垃圾价值（回收获得的空间和耗时）
优先回收价值最大的 Region（Garbage First）

混合回收：当老年代占比达到阈值时，同时回收新生代和部分老年代
```

**ZGC 特性：**
- 停顿时间 < 10ms（JDK 16+ < 1ms）
- 支持 8TB~16TB 堆
- 着色指针 + 读屏障实现并发移动对象

### 5. 内存泄漏和内存溢出的区别及排查方法

**面试官期望答案：**

> **内存泄漏**（Memory Leak）是指对象不再使用但无法被 GC 回收，逐渐耗尽内存。**内存溢出**（OOM）是指 JVM 申请不到足够的内存。内存泄漏最终会导致 OOM。排查方法：jmap 查看堆内存、MAT 分析 heap dump、关注静态集合、ThreadLocal 未清理、未关闭的资源等。

**详细解析：**

| 对比 | 内存泄漏 | 内存溢出 |
|------|---------|---------|
| 定义 | 无用的对象无法被回收 | JVM 无法分配足够的内存 |
| 关系 | 内存泄漏是 OOM 的常见原因之一 | 是最终的表现结果 |
| 排查 | MAT 分析、代码审查 | OOM dump 分析 |

**常见内存泄漏场景及代码：**

```java
// 1. 静态集合持有对象引用
public class LeakExample {
    private static final Map<String, Object> CACHE = new HashMap<>();
    // 如果只 put 不 remove，CACHE 会无限增长
    
    // 解决：使用 WeakHashMap 或 LRU 策略
}

// 2. ThreadLocal 未清理
public void process() {
    ThreadLocal<byte[]> tl = new ThreadLocal<>();
    tl.set(new byte[1024 * 1024]);  // 1MB
    // 如果使用线程池，线程被复用，ThreadLocal 的值不会被 GC
    // 解决：finally 中 tl.remove();
}

// 3. 未关闭的资源
public void read() {
    try {
        InputStream is = new FileInputStream("large.file");
        // 忘记 close()
    } catch (IOException e) { }
    // 解决：try-with-resources
    // try (InputStream is = new FileInputStream("large.file")) { }
}

// 4. 内部类持有外部类引用
public class Outer {
    private byte[] data = new byte[1024 * 1024 * 10];  // 10MB
    
    public Runnable getTask() {
        return () -> {
            // Lambda 捕获了外部类引用，Outer 无法被回收
            System.out.println(data.length);
        };
    }
}

// 5. 监听器和回调未注销
listenable.registerListener(new Listener() { ... });
// 使用完后忘记 unregister
```

### 1. 线程的创建方式

**面试官期望答案：**

> 四种方式：（1）继承 Thread 类并重写 run()；（2）实现 Runnable 接口；（3）实现 Callable 接口配合 FutureTask（可以有返回值）；（4）线程池（推荐，实际开发最常用）。实现 Runnable/Callable 更灵活（Java 单继承），线程池避免频繁创建销毁线程的开销。

```java
// 方式1：继承 Thread

    Thread.sleep(1000);
    return "线程池结果";
});
String r = future.get();
executor.shutdown();
```

### 2. synchronized 和 ReentrantLock 的区别

**面试官期望答案：**

> synchronized 是 JVM 层面的关键字，自动加锁释放锁。ReentrantLock 是 API 层面的锁，需要手动 lock/unlock（务必在 finally 中 unlock）。ReentrantLock 功能更丰富：可中断、可公平、可多个 Condition、支持 tryLock。synchronized 在 JDK 6 后经过优化（偏向锁、轻量级锁），性能已接近。

**详细解析：**

| 对比项 | synchronized | ReentrantLock |
|-------|--------------|---------------|
| 实现 | JVM 层面（monitorenter/monitorexit） | API 层面（java.util.concurrent） |
| 锁释放 | 自动（退出同步块） | 手动 lock/unlock（finally！） |
| 可中断 | 不可中断 | lockInterruptibly() 可中断 |
| 公平锁 | 非公平 | 可选公平/非公平 |
| 条件变量 | 只有一个 wait/notify | 多个 Condition |
| tryLock | 不支持 | 支持超时获取 |
| 锁状态 | 无法查询 | canHoldLock() 等 |

```java
// synchronized 用法
public synchronized void method() { ... }
public void method2() {
    synchronized (this) {
        // 同步块
    }
}

// ReentrantLock 用法
private final ReentrantLock lock = new ReentrantLock();

public void method() {
    lock.lock();
    try {
        // 临界区
    } finally {
        lock.unlock();  // 必须在 finally 中释放！
    }
}

// tryLock 带超时
if (lock.tryLock(5, TimeUnit.SECONDS)) {
    try {
        // 获取锁成功
    } finally {
        lock.unlock();
    }
} else {
    // 获取锁超时，执行降级逻辑
}
```

### 3. volatile 关键字的作用及原理

**面试官期望答案：**

> volatile 保证**可见性**（一个线程修改后其他线程立即可见）和**有序性**（禁止指令重排序），但不保证原子性。原理是通过内存屏障（Memory Barrier）实现：JVM 在 volatile 读写前后插入屏障，强制从主内存读取、刷新到主内存。适用场景：状态标志位、DCL 单例、CAS 辅助。

**详细解析：**

```java
// 场景：状态标志
private volatile boolean running = true;

public void stop() {
    running = false;  // 其他线程立即可见
}

public void work() {
    while (running) {
        // 工作循环
    }
}

// 场景：双重检查锁（DCL）单例
public class Singleton {
    // 不加 volatile 可能会返回未完全初始化的对象（指令重排序）
    private static volatile Singleton instance;
    
    public static Singleton getInstance() {
        if (instance == null) {              // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {      // 第二次检查（有锁）
                    instance = new Singleton();  // 可能发生指令重排！
                }
            }
        }
        return instance;
    }
}
```

**为什么 new Singleton() 会指令重排？**

```
正常顺序：1. 分配内存 → 2. 初始化对象 → 3. 引用指向内存地址
重排后：  1. 分配内存 → 3. 引用指向内存地址 → 2. 初始化对象
                                    ↑
                          线程B可能在此处看到非null但未初始化的对象
```

**volatile 的内存屏障语义：**

```
写操作前：StoreStore 屏障 —— 确保前面的写操作不会重排到 volatile 写之后
写操作后：StoreLoad  屏障 —— 确保 volatile 写对后续读可见
读操作前：LoadLoad   屏障 —— 确保 volatile 读不会被重排到前面
读操作后：LoadStore  屏障 —— 确保后续写操作不会重排到 volatile 读之前
```

**volatile 不保证原子性（经典反例）：**

```java
volatile int count = 0;

// 10个线程各执行 1000 次 count++，最终 count ≠ 10000
// 因为 count++ 不是原子操作（读-改-写）
// 解决方案：AtomicInteger 或 synchronized
```

### 4. 线程池的核心参数及工作原理

**面试官期望答案：**

> ThreadPoolExecutor 的核心参数：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、workQueue（任务队列）、keepAliveTime（空闲线程存活时间）、handler（拒绝策略）。工作原理：新任务先交给核心线程 → 核心线程满了放入队列 → 队列满了创建新线程到最大线程数 → 都满了执行拒绝策略。

**详细解析：**

```java
public ThreadPoolExecutor(
    int corePoolSize,          // 核心线程数
    int maximumPoolSize,       // 最大线程数
    long keepAliveTime,        // 非核心线程空闲存活时间
    TimeUnit unit,             // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,        // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)
```

**工作流程：**

```
提交任务
  │
  ├─ 线程数 < corePoolSize？
  │    ├─ 是 → 创建核心线程执行
  │    └─ 否 ↓
  ├─ workQueue 未满？
  │    ├─ 是 → 放入队列等待
  │    └─ 否 ↓
  ├─ 线程数 < maximumPoolSize？
  │    ├─ 是 → 创建非核心线程执行
  │    └─ 否 ↓
  └─ 执行拒绝策略（RejectedExecutionHandler）
```

**四种拒绝策略：**

| 策略 | 行为 |
|------|------|
| AbortPolicy（默认） | 抛出 RejectedExecutionException |
| CallerRunsPolicy | 由提交任务的线程（调用者）自己执行 |
| DiscardPolicy | 直接丢弃任务 |
| DiscardOldestPolicy | 丢弃队列中最老的任务，重新提交 |

**线程池参数配置经验：**

```java
// CPU 密集型：线程数 = CPU 核数 + 1
int cpuCores = Runtime.getRuntime().availableProcessors();
ExecutorService cpuPool = new ThreadPoolExecutor(
    cpuCores + 1, cpuCores + 1,
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>()
);

// IO 密集型：线程数 = CPU 核数 * 2（或更多）
ExecutorService ioPool = new ThreadPoolExecutor(
    cpuCores * 2, cpuCores * 2,
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 队列满时降级
);

// ⚠️ 生产环境禁止使用 Executors 创建线程池
// Executors.newFixedThreadPool → 无界队列 → 可能 OOM
// Executors.newCachedThreadPool → 最大线程 Integer.MAX → 可能 OOM
```

### 5. Java 中的锁机制（乐观锁、悲观锁、自旋锁、CAS）

**面试官期望答案：**

> **悲观锁**认为冲突多，先加锁再操作（synchronized、ReentrantLock）。**乐观锁**认为冲突少，不加锁，提交时检查冲突（CAS + 版本号）。**CAS**（Compare And Swap）是乐观锁的底层实现，原子操作（CPU 指令），问题：ABA、自旋开销。**自旋锁**是忙等待而不是阻塞，适用于锁持有时间短的场景。

**详细解析：**

**CAS 原理：**

```
CAS(V, Expected, New)
V: 内存中的值
Expected: 期望值
New: 新值

if (V == Expected) {
    V = New;  // 原子操作
    return true;
} else {
    return false;  // 冲突，重试
}
```

```java
// AtomicInteger 的 CAS 实现
public class AtomicInteger {
    private volatile int value;
    
    public final int incrementAndGet() {
        return U.getAndAddInt(this, VALUE, 1) + 1;
    }
}

// Unsafe.getAndAddInt 的核心（自旋 CAS）
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);        // 读取当前值
    } while (!compareAndSwapInt(o, offset, v, v + delta));  // CAS 重试
    return v;
}
```

**ABA 问题及解决：**

```java
// ABA 问题：值从 A → B → A，CAS 认为没变，但实际变了
// 解决方案：版本号（AtomicStampedReference）

AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(100, 0);

// 期望值 100，版本号 0，改为 101，版本号 1
boolean success = ref.compareAndSet(100, 101, 0, 1);
```

### 6. AQS 原理（AbstractQueuedSynchronizer）

**面试官期望答案：**

> AQS 是 Java 并发包的基石，核心是一个 **volatile int state**（同步状态）+ **CLH 双向队列**（等待线程队列）。提供 `tryAcquire/tryRelease`（独占）和 `tryAcquireShared/tryReleaseShared`（共享）供子类实现。ReentrantLock、Semaphore、CountDownLatch 都基于 AQS。

**详细解析：**

```java
public abstract class AbstractQueuedSynchronizer {
    private volatile int state;  // 同步状态（0 表示未锁定）
    
    // CLH 双向队列
    static final class Node {
        volatile Node prev;
        volatile Node next;
        volatile Thread thread;
        volatile int waitStatus;  // SIGNAL, CANCELLED, PROPAGATE, 0
    }
    
    private transient volatile Node head;
    private transient volatile Node tail;
    
    // 独占模式
    protected boolean tryAcquire(int arg) { throw new UnsupportedOperationException(); }
    protected boolean tryRelease(int arg) { throw new UnsupportedOperationException(); }
    
    // 共享模式
    protected int tryAcquireShared(int arg) { throw new UnsupportedOperationException(); }
    protected boolean tryReleaseShared(int arg) { throw new UnsupportedOperationException(); }
}
```

**ReentrantLock 基于 AQS 的实现：**

```
lock()
  ├─ tryAcquire(1)
  │    ├─ state == 0 → CAS(0, 1) 成功 → 获取锁
  │    ├─ state == 0 → CAS 失败 → 入队等待
  │    └─ state > 0 && 当前线程 == owner → state++ → 重入
  └─ 获取失败 → addWaiter() → acquireQueued() → park()

unlock()
  └─ tryRelease(1)
       ├─ state > 1 → state--（重入释放）
       └─ state == 0 → unpark(head.next) → 唤醒下一个
```

### 7. ThreadLocal 原理及内存泄漏问题

**面试官期望答案：**

> ThreadLocal 为每个线程维护独立的变量副本，底层是 ThreadLocalMap（key 是 ThreadLocal 的弱引用，value 是强引用）。内存泄漏：ThreadLocal 对象被回收后，key 变为 null，但 value 还在（强引用），导致 value 无法回收。解决：用完后调用 `remove()`。

**详细解析：**

```java
// ThreadLocal 原理
public class ThreadLocal<T> {
    public T get() {
        Thread t = Thread.currentThread();
        ThreadLocalMap map = t.threadLocals;  // 每个线程有自己的 map
        if (map != null) {
            Entry e = map.getEntry(this);
            if (e != null) return (T) e.value;
        }
        return setInitialValue();
    }
    
    // ThreadLocalMap 的 Entry
    static class Entry extends WeakReference<ThreadLocal<?>> {
        Object value;  // value 是强引用！
        Entry(ThreadLocal<?> k, Object v) {
            super(k);  // key 是弱引用
            value = v;
        }
    }
}
```

**内存泄漏图解：**

```
Thread → ThreadLocalMap → Entry
                            ├─ key: WeakReference<ThreadLocal> ──→ ThreadLocal (可能被GC)
                            └─ value: Object (强引用，无法被GC！)
```

```java
// 正确用法
ThreadLocal<SimpleDateFormat> sdf = ThreadLocal.withInitial(
    () -> new SimpleDateFormat("yyyy-MM-dd")
);

try {
    String date = sdf.get().format(new Date());
} finally {
    sdf.remove();  // 用完后必须 remove！
}
```

> **注意**：即使调用了 `remove()`，如果线程池中的线程被复用，上一个线程的 ThreadLocal 值可能"污染"下一个任务。所以使用线程池时务必在 finally 中 `remove()`。

### 8. ConcurrentHashMap 在高并发下的 size() 实现

**面试官期望答案：**

> ConcurrentHashMap 的 size() 使用 **LongAdder 思想**：维护一个 `baseCount` + `CounterCell[]` 数组。每个线程更新时先 CAS 更新 baseCount，如果失败则随机选一个 CounterCell 进行 CAS 累加。size() = baseCount + Σ CounterCell.value。避免了所有线程竞争一个变量的性能问题。

**详细解析：**

```java
// ConcurrentHashMap 内部
transient volatile CounterCell[] counterCells;  // JDK 8+
transient volatile long baseCount;

// addCount 中的逻辑（简化）
private final void addCount(long x, boolean wasUncontended) {
    CounterCell[] cs; long b, v; int i;
    // 1. 先尝试 CAS 更新 baseCount
    if ((cs = counterCells) != null || !U.compareAndSwapLong(this, BASECOUNT, b = baseCount, s = b + x)) {
        // 2. CAS 失败，使用 CounterCell
        boolean uncontended = wasUncontended;
        if (cs == null || cs.length == 0 || cs[i = getProbe() & m] == null ||
            !uncontended || !U.compareAndSwapLong(cs[i], CELLVALUE, v = cs[i].value, v + x)) {
            fullAddCount(x, uncontended);  // 多次失败后创建更多 CounterCell
            return;
        }
    }
    // ...
}

// size 实现
public int size() {
    long n = sumCount();
    return ((n < 0L) ? 0 : (n > Integer.MAX_VALUE) ? Integer.MAX_VALUE : (int)n);
}

final long sumCount() {
    CounterCell[] cs = counterCells;
    long sum = baseCount;
    if (cs != null) {
        for (CounterCell c : cs)
            if (c != null)
                sum += c.value;
    }
    return sum;
}
```

> **对比**：JDK 7 的 ConcurrentHashMap 使用 `Segment` 分段锁，size() 需要先尝试不加锁统计，如果统计过程中有修改，则对所有 Segment 加锁再统计。JDK 8 的 CounterCell 方案在无竞争时只有一个 baseCount 变量，竞争激烈时分散到多个 CounterCell，性能更好。

### 9. Java 中的并发工具类

**面试官期望答案：**

> **CountDownLatch**：让 N 个线程等待其他线程完成（一次性的，不可重用）。**CyclicBarrier**：让 N 个线程到达屏障点后同时继续（可重用，支持回调）。**Semaphore**：控制同时访问资源的线程数（信号量，限流）。

**详细解析：**

```java
// CountDownLatch —— 主线程等待多个子线程完成
CountDownLatch latch = new CountDownLatch(3);

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        try {
            Thread.sleep(1000);
            System.out.println(Thread.currentThread().getName() + " 完成");
        } catch (InterruptedException e) { }
        finally {
            latch.countDown();  // 计数器 -1
        }
    }).start();
}

latch.await();  // 主线程等待，直到计数器为 0
System.out.println("所有线程都完成了");

// CyclicBarrier —— 多个线程到达屏障后一起继续
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    System.out.println("所有线程到达屏障，开始下一阶段！");
});

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        try {
            Thread.sleep((long)(Math.random() * 2000));
            System.out.println(Thread.currentThread().getName() + " 到达");
            barrier.await();  // 等待其他线程
            System.out.println(Thread.currentThread().getName() + " 继续");
        } catch (Exception e) { }
    }).start();
}

// Semaphore —— 限流，最多允许 N 个线程同时访问
Semaphore semaphore = new Semaphore(3);  // 最多3个并发

for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        try {
            semaphore.acquire();    // 获取许可
            System.out.println(Thread.currentThread().getName() + " 获取许可");
            Thread.sleep(2000);
        } catch (InterruptedException e) { }
        finally {
            semaphore.release();    // 释放许可
        }
    }).start();
}
```

| 工具 | 特点 | 典型场景 |
|------|------|---------|
| CountDownLatch | 一次性，不可重置 | 等待多个任务完成后再汇总 |
| CyclicBarrier | 可重用，支持回调 | 多线程分阶段协作 |
| Semaphore | 可增减许可数 | 数据库连接池限流 |

### 1. InnoDB 和 MyISAM 的区别

**面试官期望答案：**

> InnoDB 支持事务、行锁、外键、崩溃恢复（WAL），是 MySQL 5.5+ 默认引擎。MyISAM 不支持事务、只有表锁、不支持外键，但查询速度略快（无事务开销），支持全文索引。现代开发几乎都用 InnoDB。

**详细解析：**

| 对比项 | InnoDB | MyISAM |
|-------|--------|--------|
| 事务 | 支持（ACID） | 不支持 |
| 锁粒度 | 行锁 | 表锁 |
| 外键 | 支持 | 不支持 |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| MVCC | 支持 | 不支持 |
| 全文索引 | MySQL 5.6+ 支持 | 支持 |
| 存储方式 | 表空间文件（.ibd） | .MYD（数据）+ .MYI（索引） |
| COUNT(*) | 需要扫描 | 直接存储行数 |
| 适用场景 | 高并发 OLTP | 读多写少的场景（已不推荐） |

### 2. MySQL 索引类型，聚簇索引 vs 非聚簇索引

**面试官期望答案：**

> 索引类型：主键索引、唯一索引、普通索引、组合索引。**聚簇索引**（Clustered Index）的叶子节点存储完整数据行（InnoDB 的主键索引就是聚簇索引）。**非聚簇索引**（Secondary Index）的叶子节点存储主键值，需要**回表**查询完整数据。

**详细解析：**

```
聚簇索引（主键索引）B+树：
                  [30|60|...]
               /      |       \
          [10|20]   [30|40|50]  [60|70|80]
          /    \     / | \ \     /    |    \
      [数据行] [数据行] ... [数据行] [数据行] ...
       ↑ 叶子节点直接存储完整数据行

非聚簇索引（二级索引）B+树：
                  [30|60|...]
               /      |       \
          [10|20]   [30|40|50]  [60|70|80]
          /    \     / | \ \     /    |    \
       [PK=5] [PK=15] ... [PK=45] [PK=65] ...
       ↑ 叶子节点存储主键值，需要回表查询
```

> **回表查询**：通过二级索引找到主键值，再通过主键索引查找完整数据行。如果只需要索引列的数据，则不需要回表（覆盖索引）。

### 3. 覆盖索引和最左前缀匹配原则

**面试官期望答案：**

> **覆盖索引**：查询的列全部在索引中，不需要回表（Using index）。**最左前缀匹配**：组合索引 (a, b, c) 可以匹配 a、(a, b)、(a, b, c)，但不能跳过 a 直接匹配 b 或 c。索引列顺序和查询条件顺序无关（优化器会自动调整），但中间有断开就失效。

**详细解析：**

```sql
-- 创建组合索引
CREATE INDEX idx_abc ON user(name, age, city);

-- 最左前缀匹配
SELECT * FROM user WHERE name = '张三';              -- 命中 name（最左）
SELECT * FROM user WHERE name = '张三' AND age = 25; -- 命中 name, age
SELECT * FROM user WHERE name = '张三' AND age = 25 AND city = '深圳'; -- 全部命中

-- 不匹配（跳过最左列）
SELECT * FROM user WHERE age = 25;          -- 不命中 idx_abc
SELECT * FROM user WHERE city = '深圳';     -- 不命中

-- 中间断开
SELECT * FROM user WHERE name = '张三' AND city = '深圳';  -- 只命中 name，age 断了

-- 覆盖索引（无需回表）
SELECT name, age, city FROM user WHERE name = '张三' AND age = 25;  -- Using index

-- explain 验证
EXPLAIN SELECT * FROM user WHERE name = '张三';
-- key: idx_abc, Extra: NULL（需要回表）
EXPLAIN SELECT name, age FROM user WHERE name = '张三' AND age = 25;
-- key: idx_abc, Extra: Using index（覆盖索引）
```

### 4. MySQL 事务的 ACID 特性及隔离级别

**面试官期望答案：**

> ACID：原子性（要么全成功要么全失败）、一致性（从一个一致状态到另一个）、隔离性（并发事务互不影响）、持久性（提交后永久保存）。四种隔离级别：读未提交（脏读）、读已提交（不可重复读）、可重复读（幻读，MySQL 默认）、串行化（无并发问题但性能差）。

**详细解析：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | MySQL 默认 |
|---------|------|-----------|------|-----------|
| READ UNCOMMITTED | ✓ | ✓ | ✓ | ✗ |
| READ COMMITTED | ✗ | ✓ | ✓ | ✗ |
| REPEATABLE READ | ✗ | ✗ | ✓（InnoDB 通过 MVCC+间隙锁基本解决） | ✓（默认） |
| SERIALIZABLE | ✗ | ✗ | ✗ | ✗ |

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

**脏读 vs 不可重复读 vs 幻读：**

```
脏读：读到其他事务未提交的数据（可能回滚）
不可重复读：同一事务内两次读取同一行，结果不同（被其他事务修改了）
幻读：同一事务内两次查询，多出或少了行（被其他事务插入/删除了）
```

### 5. MVCC 多版本并发控制原理

**面试官期望答案：**

> MVCC 通过**隐藏列（trx_id, roll_pointer）**、**undo log** 和**Read View** 实现无锁读。每行数据有两个隐藏列：事务 ID 和回滚指针。Read View 决定某个事务能看到哪些版本。RC 级别每次 SELECT 生成新 Read View，RR 级别事务第一次 SELECT 时生成并复用。

**详细解析：**

```
每行数据隐藏列：
+----+-------+------+-------------+-----------+
| id | name  | trx_id | roll_pointer | ...      |
+----+-------+------+-------------+-----------+

undo log 链：
当前行 → undo log v2 → undo log v1 → ...
  (trx_id=100) (trx_id=90)  (trx_id=80)
```

**Read View 判断规则：**

```
Read View 维护：
- m_ids：活跃事务 ID 列表
- min_trx_id：m_ids 中的最小值
- max_trx_id：下一个将分配的事务 ID
- creator_trx_id：创建该 Read View 的事务 ID

判断 trx_id 与 Read View 的关系：
1. trx_id == creator_trx_id → 可见（自己修改的）
2. trx_id < min_trx_id → 可见（在 Read View 之前提交）
3. trx_id >= max_trx_id → 不可见（在 Read View 之后开始）
4. min_trx_id <= trx_id < max_trx_id：
   - trx_id 在 m_ids 中 → 不可见（活跃事务，未提交）
   - trx_id 不在 m_ids 中 → 可见（已提交）
```

### 6. MySQL 的锁（行锁、表锁、间隙锁、临键锁、死锁检测与解决）

**面试官期望答案：**

> InnoDB 的锁：**行锁**（锁定单行）、**表锁**（锁定整表）、**间隙锁**（锁定范围但不含记录，防止幻读）、**临键锁**（行锁+间隙锁，InnoDB 默认）。死锁：两个事务互相等待对方释放锁。检测：`SHOW ENGINE INNODB STATUS`。解决：超时（innodb_lock_wait_timeout）、死锁检测回滚（innodb_deadlock_detect）。

**详细解析：**

```sql
-- 共享锁（S锁）和排他锁（X锁）
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;  -- 加共享锁
SELECT * FROM user WHERE id = 1 FOR UPDATE;           -- 加排他锁

-- 间隙锁（Gap Lock）
-- 假设 id 有 1, 5, 10
-- SELECT ... WHERE id BETWEEN 2 AND 8 FOR UPDATE
-- 会对 (1, 5) 和 (5, 10) 加间隙锁，不允许插入 2~9 的值

-- 死锁排查
SHOW ENGINE INNODB STATUS;
-- 查看 LATEST DETECTED DEADLOCK 部分
```

**死锁示例：**

```
事务A: UPDATE user SET age = 20 WHERE id = 1;  -- 锁定 id=1
事务B: UPDATE user SET age = 30 WHERE id = 2;  -- 锁定 id=2
事务A: UPDATE user SET age = 25 WHERE id = 2;  -- 等待事务B释放 id=2
事务B: UPDATE user SET age = 35 WHERE id = 1;  -- 等待事务A释放 id=1
→ 死锁！MySQL 死锁检测器会回滚其中一个事务
```

**避免死锁的建议：**
1. 按固定顺序访问表和行
2. 大事务拆分为小事务
3. 合理使用索引（避免锁升级为表锁）
4. 设置合理的锁等待超时

### 7. SQL 优化经验：慢 SQL 排查，explain 执行计划分析

**面试官期望答案：**

> 慢 SQL 排查步骤：开启慢查询日志 → 用 mysqldumpslow 分析 → explain 分析执行计划。explain 关键字段：type（访问类型，至少到 range 级别）、key（用到的索引）、rows（扫描行数）、Extra（Using filesort/Using temporary 表示需优化）。

**详细解析：**

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过1秒记录
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- explain 分析
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age = 25;
```

**explain 关键字段解读：**

| 字段 | 说明 | 好的值 |
|------|------|-------|
| type | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
| key | 实际用到的索引 | 非空 |
| rows | 预估扫描行数 | 越少越好 |
| Extra | 额外信息 | Using index（好） / Using filesort / Using temporary（差） |

**type 值详解：**
- `ALL`：全表扫描（最差）
- `index`：全索引扫描
- `range`：索引范围扫描（WHERE age > 20）
- `ref`：非唯一索引等值查询（WHERE name = '张三'）
- `eq_ref`：唯一索引等值查询（JOIN 场景）
- `const`：主键/唯一索引等值查询，只返回一行

**常见 SQL 优化策略：**
```sql
-- 1. 避免 SELECT *
SELECT id, name, age FROM user;  -- 只查需要的列

-- 2. 避免在索引列上使用函数
-- 差：WHERE YEAR(create_time) = 2024
-- 好：WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'

-- 3. 避免隐式类型转换
-- 差：WHERE phone = 13800138000（phone是varchar，但传了int）
-- 好：WHERE phone = '13800138000'

-- 4. LIKE 左模糊不走索引
-- 差：WHERE name LIKE '%张%'
-- 好：WHERE name LIKE '张%'（左确定走索引）

-- 5. OR 条件如果涉及不同列，考虑用 UNION 替代
-- 差：WHERE name = '张三' OR age = 25
-- 好：WHERE name = '张三' UNION SELECT * FROM user WHERE age = 25

-- 6. LIMIT 深分页优化
-- 差：SELECT * FROM user LIMIT 1000000, 10
-- 好：SELECT * FROM user WHERE id > 1000000 LIMIT 10
```

### 9. 分库分表的方案及中间件

**面试官期望答案：**

> 分库分表分为垂直拆分（按业务/模块拆分）和水平拆分（按某个规则分散数据）。水平分表策略：哈希取模、范围分片、一致性哈希。中间件：ShardingSphere（Apache，生态完善）、MyCat（社区活跃度下降）。拆分后的问题：分布式事务、跨库 JOIN、全局 ID、数据迁移。

**详细解析：**

```
垂直分库：按业务拆分
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户库    │  │ 订单库    │  │ 商品库    │
│ user     │  │ order    │  │ product  │
└──────────┘  └──────────┘  └──────────┘

水平分表：同一表按规则分散
┌──────────┐  ┌──────────┐  ┌──────────┐
│ order_0  │  │ order_1  │  │ order_2  │
│ user_id%3│  │ user_id%3│  │ user_id%3│
│  = 0     │  │  = 1     │  │  = 2     │
└──────────┘  └──────────┘  └──────────┘
```

```yaml
# ShardingSphere 配置示例（YAML）
rules:
  - !SHARDING
    tables:
      t_order:
        actualDataNodes: ds_${0..1}.t_order_${0..1}
        tableStrategy:
          standard:
            shardingColumn: order_id
            shardingAlgorithmName: t_order_inline
    shardingAlgorithms:
      t_order_inline:
        type: INLINE
        props:
          algorithm-expression: t_order_${(order_id % 2).intValue()}
```

**分库分表带来的问题：**
1. 分布式事务（Seata 解决）
2. 跨库 JOIN（数据冗余、全局表）

### 10. 主从复制原理及读写分离

**面试官期望答案：**

> 主从复制：主库将变更写入 **binlog**，从库的 IO 线程拉取 binlog 写入 relay log，SQL 线程重放 relay log 实现数据同步。读写分离：写操作走主库，读操作走从库，通过中间件（ShardingSphere、MyCat、ProxySQL）或代码层面实现。问题：主从延迟。

**详细解析：**

```
主库                              从库
┌──────────┐                   ┌──────────────┐
│ Client    │                   │ IO Thread    │
│  写入     │→ binlog ──────→  │  拉取binlog   │
│           │                   ├──────────────┤
│ binlog    │                   │ relay log    │
│ dump      │                   ├──────────────┤
│           │                   │ SQL Thread   │
│           │                   │  重放relay log│
└──────────┘                   └──────────────┘
```

```sql
-- 主从复制配置（简述）
-- 主库 my.cnf:
server-id = 1
log-bin = mysql-bin
binlog_format = ROW

-- 从库:
server-id = 2
relay-log = relay-bin
read-only = 1

-- 从库连接主库
CHANGE MASTER TO
    MASTER_HOST='master_ip',
    MASTER_USER='repl',
    MASTER_PASSWORD='password',
    MASTER_LOG_FILE='mysql-bin.000001',
    MASTER_LOG_POS=0;

START SLAVE;
```

**解决主从延迟：**
1. 强制读主（关键数据）
2. 并行复制（slave_parallel_workers）
3. 半同步复制（semi-sync，确保至少一个从库接收）
4. 降低单次大事务

### 2. Redis 持久化机制（RDB、AOF）及选择策略

**面试官期望答案：**

> **RDB**（快照）：定期将内存数据保存为二进制文件，恢复快但可能丢失数据。**AOF**（追加日志）：记录每个写命令，数据更安全但文件大、恢复慢。Redis 4.0+ 支持**混合持久化**：RDB 做基础快照 + 增量 AOF。生产环境推荐 AOF 或混合持久化。

**详细解析：**

```bash
# RDB 配置
save 900 1      # 900秒内有1次修改则触发
save 300 10     # 300秒内有10次修改
save 60 10000   # 60秒内有10000次修改
dbfilename dump.rdb
dir /var/lib/redis

# 手动触发
SAVE    # 同步阻塞（生产慎用）
BGSAVE  # 异步fork子进程

# AOF 配置
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec  # 每秒刷盘（推荐）

# 混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes
```

**RDB vs AOF 对比：**

| 对比项 | RDB | AOF |
|-------|-----|-----|
| 数据安全 | 可能丢失（取决于 save 频率） | 最多丢失 1 秒（everysec） |
| 文件大小 | 小（二进制压缩） | 大（记录所有写命令） |
| 恢复速度 | 快 | 慢（需要重放命令） |
| 性能影响 | fork 时有瞬间阻塞 | 每秒写磁盘（everysec） |
| 适用场景 | 可容忍数据丢失的缓存 | 需要数据安全的持久化 |

### 3. Redis 缓存穿透、缓存击穿、缓存雪崩的解决方案

**面试官期望答案：**

> **缓存穿透**：查询不存在的数据，缓存和 DB 都没有。解决：布隆过滤器 + 缓存空值。**缓存击穿**：热点 key 过期瞬间大量请求打到 DB。解决：互斥锁 + 永不过期（逻辑过期）。**缓存雪崩**：大量 key 同时过期。解决：过期时间加随机数 + 多级缓存 + 持久缓存。

**详细解析：**

```java
// 缓存穿透 —— 布隆过滤器 + 缓存空值
public Object getWithBloomFilter(String key) {
    // 1. 布隆过滤器判断（不存在则一定不存在）
    if (!bloomFilter.mightContain(key)) {
        return null;  // 直接返回，不查DB
    }
    
    // 2. 查缓存
    String value = redis.get(key);
    if (value != null) {
        return deserialize(value);
    }
    
    // 3. 查 DB
    Object result = db.query(key);
    if (result == null) {
        // 缓存空值，设置短过期时间
        redis.setex(key, 60, "NULL");
    } else {
        redis.setex(key, 3600, serialize(result));
    }
    return result;
}

// 缓存击穿 —— 互斥锁（setNX）
public Object getWithLock(String key) {
    String value = redis.get(key);
    if (value != null) {
        return deserialize(value);
    }
    
    // 加锁，防止并发穿透
    String lockKey = "lock:" + key;
    if (redis.setnx(lockKey, "1", 10, TimeUnit.SECONDS)) {
        try {
            // double check
            value = redis.get(key);
            if (value != null) return deserialize(value);
            
            Object result = db.query(key);
            redis.setex(key, 3600, serialize(result));
            return result;
        } finally {
            redis.del(lockKey);
        }
    } else {
        // 未获取锁，短暂等待后重试
        Thread.sleep(100);
        return getWithLock(key);
    }
}
```

### 4. Redis 分布式锁的实现

**面试官期望答案：**

> 基本实现：`SET key value NX EX seconds`（原子操作）。问题：锁超时后业务没执行完导致锁被其他线程获取 → 续期（看门狗，如 Redisson）。Redisson 是生产级方案：自动续期、可重入、红锁（多节点）。

**详细解析：**

```java
// 基本实现
public boolean tryLock(String lockKey, String requestId, int expireSeconds) {
    // SET lockKey requestId NX EX expireSeconds（原子操作）
    String result = jedis.set(lockKey, requestId, "NX", "EX", expireSeconds);
    return "OK".equals(result);
}

public boolean releaseLock(String lockKey, String requestId) {
    // Lua 脚本保证原子性（判断+删除）
    String script = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "   return redis.call('del', KEYS[1]) " +
        "else " +
        "   return 0 " +
        "end";
    Object result = jedis.eval(script, Collections.singletonList(lockKey), 
                               Collections.singletonList(requestId));
    return Long.valueOf(1L).equals(result);
}

// Redisson 实现（推荐）
RLock lock = redisson.getLock("myLock");
try {
    lock.lock(30, TimeUnit.SECONDS);  // 自动续期（看门狗默认30秒）
    // 业务逻辑
} finally {
    lock.unlock();
}
```

> **面试追问**：Redisson 的看门狗机制 — 默认 leaseTime = 30s，后台线程每隔 10s（30/3）检查锁是否被持有，如果是则自动续期到 30s。如果业务执行完调用 unlock()，看门狗自动停止。

### 5. Redis 的内存淘汰策略

**面试官期望答案：**

> Redis 8 种淘汰策略（maxmemory-policy）：noeviction（默认，不淘汰）、allkeys-lru（所有 key 中淘汰最近最少使用的）、allkeys-lfu（最不经常使用的）、allkeys-random（随机）、volatile-lru（过期 key 中 LRU）、volatile-lfu、volatile-random、volatile-ttl（淘汰即将过期的）。生产推荐 allkeys-lru 或 allkeys-lfu。

**详细解析：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| noeviction | 不淘汰，写入报错 | 数据不能丢失 |
| allkeys-lru | 所有 key 中 LRU | 通用缓存 |
| allkeys-lfu | 所有 key 中 LFU（Redis 4.0+） | 热点数据缓存 |
| volatile-lru | 过期 key 中 LRU | 只淘汰设置了过期时间的 |
| volatile-lfu | 过期 key 中 LFU | 同上 |
| volatile-ttl | 淘汰 TTL 最短的 | 有过期策略的缓存 |
| allkeys-random | 所有 key 随机淘汰 | — |
| volatile-random | 过期 key 随机淘汰 | — |

```bash

# 查看
INFO memory
```

### 9. Redis 的过期策略

**面试官期望答案：**

> Redis 使用**惰性删除 + 定期删除**双策略。**惰性删除**：访问 key 时检查是否过期，过期则删除。**定期删除**：每秒执行 10 次检查，每次随机抽查一批 key，过期则删除，如果过期比例 > 25% 则继续检查。配合内存淘汰策略（LRU/LFU/TTL），确保内存不会用满。

### 3. Spring 的事务传播机制

**面试官期望答案：**

> **REQUIRED**（默认）：有事务则加入，无则新建。**REQUIRES_NEW**：总是新建事务，挂起当前事务。**NESTED**：嵌套事务（保存点）。**SUPPORTS**：有事务则加入，无则非事务执行。**NOT_SUPPORTED**：非事务执行，挂起当前事务。**MANDATORY**：必须在事务中，否则抛异常。**NEVER**：不能在事务中，否则抛异常。

**详细解析：**

```java
@Service
public class OrderService {
    
    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder() {
        orderDao.insert(order);        // 在当前事务中
        logService.addLog();            // REQUIRED → 加入当前事务
        inventoryService.deduct();     // REQUIRES_NEW → 新事务，独立提交
    }
}

@Service
public class InventoryService {
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void deduct() {
        inventoryDao.update(stock);
        // 这个事务独立于外层事务
        // 即使外层回滚，这里也提交了
    }
}
```

| 传播行为 | 当前有事务 | 当前无事务 | 典型场景 |
|---------|----------|----------|---------|
| REQUIRED | 加入 | 新建 | 大多数业务方法（默认） |
| REQUIRES_NEW | 新建（挂起当前） | 新建 | 独立事务（如日志、通知） |
| NESTED | 嵌套（保存点） | 新建 | 子操作回滚不影响外层 |
| SUPPORTS | 加入 | 非事务 | 查询方法 |
| NOT_SUPPORTED | 挂起 | 非事务 | 非事务操作 |

### 7. Spring 中的循环依赖如何解决？

**面试官期望答案：**

> Spring 通过**三级缓存**解决单例 Bean 的循环依赖（构造器注入无法解决）：一级缓存（singletonObjects，完全初始化的 Bean）、二级缓存（earlySingletonObjects，早期引用）、三级缓存（singletonFactories，ObjectFactory）。A 依赖 B，B 依赖 A 时，A 先创建，将工厂放入三级缓存；B 创建时从三级缓存获取 A 的早期引用，完成初始化后双方都放入一级缓存。

**详细解析：**

```java
// 三级缓存
/** 一级缓存：完全初始化的 Bean */
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

/** 二级缓存：早期 Bean 引用（已实例化但未初始化） */
private final Map<String, Object> earlySingletonObjects = new HashMap<>(16);

/** 三级缓存：ObjectFactory（用于产生早期引用，可创建 AOP 代理） */
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
```

**解决流程：**

```
A 实例化 → 放入三级缓存（A 的 ObjectFactory）
  ↓
A 属性注入需要 B
  ↓
B 实例化 → 放入三级缓存（B 的 ObjectFactory）
  ↓
B 属性注入需要 A → 从三级缓存获取 A 的早期引用 → 放入二级缓存
  ↓
B 初始化完成 → 从二级/三级移到一级缓存
  ↓
A 属性注入完成（B 已在一级缓存）
  ↓
A 初始化完成 → 从二级/三级移到一级缓存
```

> **注意**：构造器注入的循环依赖**无法解决**，因为实例化阶段就需要依赖对象。解决方法：改用字段注入（@Autowired）或使用 `@Lazy`。

### 8. @Transactional 失效的场景

**面试官期望答案：**

> 常见失效场景：（1）方法不是 public；（2）同类方法内部调用（没有经过代理）；（3）异常被 catch 未抛出；（4）抛出非 RuntimeException（checked exception 未 rollbackFor）；（5）数据库引擎不支持事务（MyISAM）；（6）未开启事务管理器。

**详细解析：**

```java
@Service
public class OrderService {
    
    // ❌ 同类内部调用，@Transactional 失效
    public void methodA() {
        this.methodB();  // 直接调用，不经过代理
    }
    
    @Transactional
    public void methodB() {
        // 事务不生效！
    }
    
    // ✅ 解决：注入自身或使用 AopContext
    @Autowired
    private OrderService self;
    
    public void methodA2() {
        self.methodB();  // 经过代理，事务生效
    }
    
    // ❌ 异常被 catch 未抛出
    @Transactional
    public void methodC() {
        try {
            // 业务逻辑，抛出异常
        } catch (Exception e) {
            log.error("出错了", e);  // 吞了异常，事务不回滚！
        }
    }
    
    // ✅ 解决：catch 后手动回滚
    @Transactional
    public void methodC2() {
        try {
            // 业务逻辑
        } catch (Exception e) {
            log.error("出错了", e);
            TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        }
    }
    
    // ❌ checked exception 默认不回滚
    @Transactional
    public void methodD() throws IOException {
        throw new IOException("checked exception");  // 不回滚！
    }
    
    // ✅ 解决：指定 rollbackFor
    @Transactional(rollbackFor = Exception.class)
    public void methodD2() throws IOException {
        throw new IOException("checked exception");  // 回滚
    }
}
```

**面试官期望答案：**

### 3. 分布式系统的 CAP 定理和 BASE 理论

**面试官期望答案：**

### 8. 分布式 ID 生成方案

**面试官期望答案：**

> **雪花算法**（Snowflake）：64 bit = 1bit 符号 + 41bit 时间戳 + 10bit 机器 ID + 12bit 序列号，趋势递增、高性能。**UUID**：简单但无序、太长。**数据库自增**：简单但有单点瓶颈。**号段模式**（Leaf）：数据库预分配号段，减少数据库访问。推荐雪花算法或 Leaf。

**详细解析：**

```java
// 雪花算法结构（64 bit）：
// 0 | 41 bit 时间戳（毫秒，约69年）| 10 bit 机器ID（5位数据中心+5位机器）| 12 bit 序列号
// 最大支持 4096 个节点，每个节点每毫秒最多生成 4096 个 ID

public class SnowflakeIdGenerator {
    private final long epoch = 1609459200000L;  // 2021-01-01
    
    private final long datacenterIdBits = 5L;
    private final long workerIdBits = 5L;
    private final long sequenceBits = 12L;
    
    private long lastTimestamp = -1L;
    private long sequence = 0L;
    
    private final long datacenterId;
    private final long workerId;
    
    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("时钟回拨");
        }
        
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & (1L << sequenceBits - 1);
            if (sequence == 0) {
                timestamp = tilNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        
        lastTimestamp = timestamp;
        
        return ((timestamp - epoch) << (datacenterIdBits + workerIdBits + sequenceBits))
             | (datacenterId << (workerIdBits + sequenceBits))
             | (workerId << sequenceBits)
             | sequence;
    }
    
    private long tilNextMillis(long lastTimestamp) {
        long timestamp = System.currentTimeMillis();
        while (timestamp <= lastTimestamp) {
            timestamp = System.currentTimeMillis();
        }
        return timestamp;
    }
}
```

### 2. 手写：快速排序算法及时间复杂度分析

```java
public class QuickSort {
    public static void sort(int[] arr, int low, int high) {
        if (low < high) {
            int pivotIndex = partition(arr, low, high);
            sort(arr, low, pivotIndex - 1);
            sort(arr, pivotIndex + 1, high);
        }
    }
    
    private static int partition(int[] arr, int low, int high) {
        int pivot = arr[high];  // 选最后一个元素为基准
        int i = low - 1;        // i 是小于 pivot 的区域的右边界
        
        for (int j = low; j < high; j++) {
            if (arr[j] <= pivot) {
                i++;
                swap(arr, i, j);
            }
        }
        swap(arr, i + 1, high);  // 将 pivot 放到正确位置
        return i + 1;
    }
    
    private static void swap(int[] arr, int i, int j) {
        int temp = arr[i];
        arr[i] = arr[j];
        arr[j] = temp;
    }
    
    public static void main(String[] args) {
        int[] arr = {5, 3, 8, 4, 2, 7, 1, 10};
        sort(arr, 0, arr.length - 1);
        System.out.println(Arrays.toString(arr));  // [1, 2, 3, 4, 5, 7, 8, 10]
    }
}
```

**时间复杂度：**
- 最优：O(n log n)（每次 pivot 都将数组等分）
- 平均：O(n log n)
- 最差：O(n²)（数组已有序，每次 pivot 是最大/最小值）
- **空间复杂度**：O(log n)（递归栈深度）

### 3. 手写：二分查找算法

```java
public class BinarySearch {
    // 标准二分查找（找到目标返回索引，没找到返回 -1）
    public static int search(int[] arr, int target) {
        int left = 0, right = arr.length - 1;
        
        while (left <= right) {
            int mid = left + (right - left) / 2;  // 防止溢出
            
            if (arr[mid] == target) {
                return mid;
            } else if (arr[mid] < target) {
                left = mid + 1;
            } else {
                right = mid - 1;
            }
        }
        
        return -1;
    }
    
    public static void main(String[] args) {
        int[] arr = {1, 3, 5, 7, 9, 11, 13};
        System.out.println(search(arr, 7));   // 输出: 3
        System.out.println(search(arr, 6));   // 输出: -1
    }
}
```

> **注意**：`mid = left + (right - left) / 2` 而不是 `(left + right) / 2`，防止 `left + right` 整数溢出。

### 5. LRU Cache 的实现思路

**面试官期望答案：**

> LRU（Least Recently Used）缓存淘汰策略：淘汰最近最少使用的数据。实现方式：**HashMap + 双向链表**。HashMap 提供 O(1) 查找，双向链表维护访问顺序（访问时移到头部，淘汰时移除尾部）。

**详细解析：**

```java
public class LRUCache<K, V> {
    
    private final int capacity;
    private final Map<K, Node> map = new HashMap<>();
    private final Node head = new Node();  // 哨兵头
    private final Node tail = new Node();  // 哨兵尾
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }
    
    public V get(K key) {
        Node node = map.get(key);
        if (node == null) return null;
        moveToHead(node);  // 访问后移到头部
        return node.value;
    }
    
    public void put(K key, V value) {
        Node node = map.get(key);
        if (node != null) {
            node.value = value;
            moveToHead(node);
        } else {
            node = new Node(key, value);
            map.put(key, node);
            addToHead(node);
            if (map.size() > capacity) {
                Node removed = removeTail();
                map.remove(removed.key);
            }
        }
    }
    
    private void addToHead(Node node) {
        node.prev = head;
        node.next = head.next;
        head.next.prev = node;
        head.next = node;
    }
    
    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }
    
    private void moveToHead(Node node) {
        removeNode(node);
        addToHead(node);
    }
    
    private Node removeTail() {
        Node node = tail.prev;
        removeNode(node);
        return node;
    }
    
    class Node {
        K key;
        V value;
        Node prev, next;
        Node() {}
        Node(K key, V value) {
            this.key = key;
            this.value = value;
        }
    }
}
```

> **面试追问**：为什么用双向链表而不是单向链表？  
> 答：删除节点时需要 O(1) 操作，双向链表可以直接通过 `node.prev.next = node.next` 删除，单向链表需要遍历找到前驱节点。

### 7. 二叉树的遍历

```java
public class TreeTraversal {
    static class TreeNode {
        int val;
        TreeNode left, right;
        TreeNode(int val) { this.val = val; }
    }
    
    // 前序遍历：根 → 左 → 右
    public static void preOrder(TreeNode root) {
        if (root == null) return;
        System.out.print(root.val + " ");
        preOrder(root.left);
        preOrder(root.right);
    }
    
    // 中序遍历：左 → 根 → 右
    public static void inOrder(TreeNode root) {
        if (root == null) return;
        inOrder(root.left);
        System.out.print(root.val + " ");
        inOrder(root.right);
    }
    
    // 后序遍历：左 → 右 → 根
    public static void postOrder(TreeNode root) {
        if (root == null) return;
        postOrder(root.left);
        postOrder(root.right);
        System.out.print(root.val + " ");
    }
    
    // 层序遍历（BFS）
    public static void levelOrder(TreeNode root) {
        if (root == null) return;
        Queue<TreeNode> queue = new LinkedList<>();
        queue.offer(root);
        
        while (!queue.isEmpty()) {
            TreeNode node = queue.poll();
            System.out.print(node.val + " ");
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
    }
}
```

```java
// 0-1 背包问题
public class Knapsack {
    public static int knapsack(int[] weight, int[] value, int capacity) {
        int n = weight.length;
        int[] dp = new int[capacity + 1];
        
        for (int i = 0; i < n; i++) {
            // 逆序遍历，防止重复使用同一物品
            for (int w = capacity; w >= weight[i]; w--) {
                dp[w] = Math.max(dp[w], dp[w - weight[i]] + value[i]);
            }
        }
        
        return dp[capacity];
    }
    
    public static void main(String[] args) {
        int[] weight = {2, 3, 4, 5};
        int[] value = {3, 4, 5, 6};
        int capacity = 8;
        System.out.println(knapsack(weight, value, capacity));  // 输出: 10
    }
}

// 最长公共子序列（LCS）
public class LCS {
    public static int lcs(String s1, String s2) {
        int m = s1.length(), n = s2.length();
        int[][] dp = new int[m + 1][n + 1];
        
        for (int i = 1; i <= m; i++) {
            for (int j = 1; j <= n; j++) {
                if (s1.charAt(i - 1) == s2.charAt(j - 1)) {
                    dp[i][j] = dp[i - 1][j - 1] + 1;
                } else {
                    dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);
                }
            }
        }
        
        return dp[m][n];
    }
    
    public static void main(String[] args) {
        System.out.println(lcs("ABCBDAB", "BDCAB"));  // 输出: 4 ("BCAB")
    }
}
```

### 9. 手写：生产者-消费者模式

```java
public class ProducerConsumer {
    private final BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(10);
    
    // 生产者
    class Producer implements Runnable {
        @Override
        public void run() {
            for (int i = 1; i <= 20; i++) {
                try {
                    queue.put(i);  // 队列满时自动阻塞
                    System.out.println(Thread.currentThread().getName() + " 生产: " + i);
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }
        }
    }
    
    // 消费者
    class Consumer implements Runnable {
        @Override
        public void run() {
            while (true) {
                try {
                    int item = queue.take();  // 队列空时自动阻塞
                    System.out.println(Thread.currentThread().getName() + " 消费: " + item);
                    Thread.sleep(200);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        }
    }
    
    public static void main(String[] args) {
        ProducerConsumer pc = new ProducerConsumer();
        ExecutorService executor = Executors.newFixedThreadPool(4);
        executor.submit(pc.new Producer());
        executor.submit(pc.new Producer());
        executor.submit(pc.new Consumer());
        executor.submit(pc.new Consumer());
        executor.shutdown();
    }
}
```

> **BlockingQueue 的原理**：内部使用 ReentrantLock + Condition（notFull、notEmpty）实现等待/通知机制。put() 时队列满则 notFull.await()，take() 时队列空则 notEmpty.await()。

```
              └───────────┘    └─────────────┘   └─────────────┘
```

```java
// 秒杀核心逻辑
@Service
public class SeckillService {
    
    @Autowired
    private RedisTemplate redisTemplate;
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    public SeckillResult seckill(Long userId, Long productId) {
        // 1. 限流：用户维度（每人限购1次）
        Boolean isNew = redisTemplate.opsForValue()
            .setIfAbsent("seckill:user:" + userId + ":" + productId, "1", 1, TimeUnit.DAYS);
        if (Boolean.FALSE.equals(isNew)) {
            return SeckillResult.fail("您已参与过");
        }
        
        // 2. Redis 预扣库存（Lua 脚本保证原子性）
        String script = 
            "local stock = tonumber(redis.call('GET', KEYS[1])) " +
            "if stock > 0 then " +
            "   redis.call('DECR', KEYS[1]) " +
            "   return 1 " +
            "else " +
            "   return 0 " +
            "end";
        
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList("seckill:stock:" + productId)
        );
        
        if (result == null || result == 0) {
            return SeckillResult.fail("库存不足");
        }
        
        // 3. 发送 MQ 异步创建订单
        SeckillMessage msg = new SeckillMessage(userId, productId);
        mqTemplate.convertAndSend("seckill-topic", msg);
        
        return SeckillResult.success("排队中，请等待结果");
    }
}
```

**关键措施：**
1. 前端：按钮防重复点击 + CDN 静态化
2. 网关：Sentinel 限流（QPS 限制）
3. Redis：Lua 脚本原子扣库存
4. MQ：异步下单，削峰填谷
5. 数据库：乐观锁防超卖（`UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0`）
6. 降级：库存不足时直接返回失败

### 2. 如何设计一个简单的分布式爬虫？

**核心设计：**
- **调度中心**（Redis）：URL 队列 + 去重集合
- **Worker 节点**：从队列取 URL，抓取，解析，新 URL 入队
- **存储**：抓取结果写入数据库/ES
- **去重**：Redis Set 或布隆过滤器

```java
// 爬虫核心逻辑
@Service
public class CrawlerService {
    
    @Autowired
    private RedisTemplate redisTemplate;
    @Autowired
    private RestTemplate restTemplate;
    @Autowired
    private ElasticsearchRestTemplate esTemplate;
    
    public void crawl(String startUrl) {
        redisTemplate.opsForSet().add("crawler:todo", startUrl);
    }
    
    @Scheduled(fixedDelay = 100)
    public void process() {
        // 从队列取出 URL
        String url = (String) redisTemplate.opsForSet().pop("crawler:todo");
        if (url == null) return;
        
        // 去重检查
        Boolean isNew = redisTemplate.opsForSet().add("crawler:visited", url);
        if (Boolean.FALSE.equals(isNew)) return;
        
        // 抓取页面
        String html = restTemplate.getForObject(url, String.class);
        
        // 解析页面，提取新链接
        List<String> links = parseLinks(html);
        for (String link : links) {
            redisTemplate.opsForSet().add("crawler:todo", link);
        }
        
        // 存储内容
        saveToEs(url, html);
    }
}
```

```bash
# - 内存泄漏（长生命周期对象持有短生命周期引用）
# - 一次性加载大文件/数据
```

### 4. Docker 和 K8s 的基本概念

**Docker：**
```bash
# 基本命令
docker build -t myapp:1.0 .           # 构建镜像
docker run -d -p 8080:8080 myapp:1.0  # 运行容器
docker logs -f <container_id>         # 查看日志
docker exec -it <container_id> bash   # 进入容器

# Dockerfile 示例
FROM openjdk:17-jdk-slim
WORKDIR /app
COPY target/myapp.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**Kubernetes 核心概念：**
- **Pod**：最小部署单元（1 个或多个容器）
- **Service**：服务发现和负载均衡
- **Deployment**：管理 Pod 的副本数、滚动更新
- **ConfigMap/Secret**：配置和密钥管理
- **Ingress**：外部流量入口

```yaml
# K8s Deployment 示例
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:1.0
        ports:
        - containerPort: 8080
```

### 5. CI/CD 流程

**Jenkins Pipeline 基本流程：**

```
代码提交 → Git Hook 触发 → Jenkins 拉取代码 → 编译构建 → 单元测试
→ 代码质量检查（SonarQube）→ 构建镜像 → 推送到镜像仓库 → 部署到测试环境
→ 自动化测试 → 审批 → 部署到生产环境
```

```groovy
// Jenkinsfile 示例
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }
        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }
        stage('Docker Build') {
            steps {
                sh 'docker build -t myapp:${BUILD_NUMBER} .'
            }
        }
        stage('Deploy') {
            steps {
                sh 'kubectl apply -f k8s/'
            }
        }
    }
}
```
