## 一、Java 基础与 JVM

### 1. JVM 运行时数据区有哪些？JDK 8 之后有什么变化？

**【阿里/腾讯/字节高频考点】**

**答：**

JVM 运行时数据区主要分为以下几部分：

**线程私有区域：**
- **程序计数器（PC Register）**：记录当前线程所执行的字节码行号指示器，唯一不会发生 OOM 的区域。执行 native 方法时值为 Undefined。
- **虚拟机栈（JVM Stack）**：每个方法执行时创建一个栈帧，存储局部变量表、操作数栈、动态链接、方法返回地址。超过最大深度抛出 `StackOverflowError`，动态扩展失败时抛 `OOM`。
- **本地方法栈（Native Method Stack）**：为 Native 方法服务，HotSpot 中与虚拟机栈合二为一。

**线程共享区域：**
- **堆（Heap）**：最大的内存区域，存放对象实例。分为新生代（Eden + Survivor×2）和老年代。GC 的主战场。
- **方法区（Method Area）**：存储类信息、常量、静态变量、JIT 编译后的代码。

**JDK 7 → JDK 8 核心变化：**
| 变更内容 | JDK 7 | JDK 8 |
|--------|--------|--------|
| 方法区实现 | 永久代（PermGen），固定大小上限 | 元空间（Metaspace），使用本地内存 |
| 字符串常量池 | 方法区 | 堆 |
| 静态变量 | 方法区 | 堆（与 Class 对象一起存放） |

**为什么用元空间替换永久代？**
1. 永久代大小固定，容易 OOM（`java.lang.OutOfMemoryError: PermGen space`）
2. 元空间使用本地内存，受系统内存限制，可配置 `-XX:MaxMetaspaceSize`
3. 合并 HotSpot 和 JRockit 代码时，JRockit 没有永久代概念

### 2. 对象创建的过程是什么？

**【华为/美团高频】**

**答：** 分五步：

1. **类加载检查**：检查常量池中是否有该类的符号引用，若未加载则先执行类加载。
2. **分配内存**：
   - 堆规整时用**指针碰撞**（Serial、ParNew 收集器）
   - 堆不规整时用**空闲列表**（CMS 收集器）
   - 线程安全保障：CAS + 失败重试，或 TLAB（Thread Local Allocation Buffer）
3. **初始化零值**：将分配到的内存空间初始化为零值（不含对象头）
4. **设置对象头**：设置类的元数据指针、GC 分代年龄、锁状态标志、哈希码等
5. **执行 `<init>` 方法**：按程序员意愿初始化对象

### 3. 什么是垃圾回收？常用 GC 算法有哪些？

**【阿里/字节/京东高频】**

**答：**

**判断对象可回收的方法：**
- **引用计数法**：存在循环引用问题，Java 不使用
- **可达性分析算法**：从 GC Roots 出发，不可达的对象可被回收。GC Roots 包括：虚拟机栈中引用的对象、方法区中静态变量引用的对象、本地方法栈 JNI 引用的对象

**三大基础 GC 算法：**

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|--------|
| 标记-清除（Mark-Sweep） | 简单 | 内存碎片，执行效率不高 | CMS 老年代 |
| 复制（Copying） | 无碎片，效率高 | 内存利用率低（50%） | 新生代 Eden→Survivor |
| 标记-整理（Mark-Compact） | 无碎片 | 移动对象，需更新引用 | 老年代 |

**常用 GC 收集器：**
- **Serial/Serial Old**：单线程，Stop-The-World，适合客户端
- **ParNew**：Serial 的多线程版，配合 CMS 使用
- **Parallel Scavenge/Parallel Old**：注重吞吐量，适合后台计算
- **CMS（Concurrent Mark Sweep）**：并发低停顿，但有内存碎片、浮动垃圾问题
- **G1（Garbage First）**：JDK 9 默认。将堆划分为等大 Region，混合收集，可预测停顿时间
- **ZGC**：JDK 11 引入，几乎全程并发，停顿时间 < 10ms，支持 TB 级堆

**G1 vs CMS 核心对比：**
- G1 无内存碎片（整理），CMS 有
- G1 可以设置期望停顿时间（`-XX:MaxGCPauseMillis`）
- G1 占用更多内存和 CPU

### 4. 说说类加载机制和双亲委派模型

**【阿里/腾讯/华为架构师必考】**

**答：**

**类加载过程（7步）：**
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载

关键阶段说明：
- **准备**：为静态变量分配内存并赋默认零值（`static int x = 5` 此阶段 x=0）
- **初始化**：执行 `<clinit>()` 方法，赋真正的初始值（x=5）

**双亲委派模型：**

类加载器层次：Bootstrap ClassLoader（C++实现）→ Extension ClassLoader → Application ClassLoader → 自定义 ClassLoader

**工作原理：** 收到类加载请求时，先委托父加载器加载，所有父加载器都无法加载时，自己才尝试加载。

**作用：**
1. 防止核心 API 被篡改（如自定义 `java.lang.String` 不会生效）
2. 保证同一个类只被加载一次，确保类的唯一性

**如何打破双亲委派？**
- 重写 `ClassLoader` 的 `loadClass()` 方法（而不是 `findClass()`）
- 实例：JDBC、Tomcat（Web 应用隔离）、OSGi（模块热部署）、Spring Boot devtools

### 5. JVM 调优常用参数有哪些？如何排查 OOM？

**【阿里P7/腾讯T8架构师面试真题】**

**答：**

**常用 JVM 参数：**
```bash
# 堆大小
-Xms4g -Xmx4g           # 初始/最大堆大小（建议相同，避免扩容）
-Xmn2g                   # 新生代大小
-XX:MetaspaceSize=256m   # 元空间初始大小
-XX:MaxMetaspaceSize=512m

# GC 选择
-XX:+UseG1GC             # 使用 G1（JDK 9+ 默认）
-XX:MaxGCPauseMillis=200 # G1 目标停顿时间
-XX:+UseZGC              # 使用 ZGC（JDK 15+ 正式）

# GC 日志
-Xlog:gc*:file=gc.log:time,level,tags

# OOM 时导出堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heapdump.hprof
```

**OOM 排查步骤：**
1. 查看 GC 日志，确认是哪种 OOM（堆/方法区/直接内存）
2. 使用 `jmap -dump:live,format=b,file=heap.hprof <pid>` 导出堆
3. 用 **MAT（Eclipse Memory Analyzer）** 或 **VisualVM** 分析堆转储
4. 找到 Retained Heap 最大的对象，分析引用链
5. 常见原因：内存泄漏（集合没有清除）、大对象、连接/线程池未释放

## 二、并发编程

### 6. synchronized 底层原理是什么？锁的升级过程如何？

**【阿里/字节/拼多多高频考点】**

**答：**

**底层实现：**
- **同步代码块**：基于 `monitorenter` 和 `monitorexit` 字节码指令，依赖对象的 Monitor（监视器锁）
- **同步方法**：通过方法 flags 中的 `ACC_SYNCHRONIZED` 标志实现

每个对象有一个关联的 Monitor，Monitor 依赖操作系统的 `Mutex Lock`（互斥量）实现，线程切换需要从用户态切换到内核态，开销较大。

**锁升级（无锁 → 偏向锁 → 轻量级锁 → 重量级锁）：**

| 状态 | 触发条件 | 存储位置 | 性能 |
|------|---------|---------|------|
| 无锁 | 初始状态 | Mark Word | 最高 |
| 偏向锁 | 只有一个线程访问（JDK 15 已默认关闭） | Mark Word 存线程ID | 高 |
| 轻量级锁 | 多线程竞争但不激烈，CAS 自旋 | 栈帧中的锁记录 | 中 |
| 重量级锁 | 自旋失败，线程阻塞挂起 | Monitor 对象 | 低 |

**JDK 6 引入的锁优化：**
- **锁消除**：JIT 编译时，对检测到不可能存在竞争的锁进行消除（如方法内的局部 StringBuilder）
- **锁粗化**：多个连续加锁/解锁合并为一次（如循环内 synchronized）
- **适应性自旋**：自旋次数根据历史自旋成功率动态调整

> 注意：JDK 15 偏向锁默认关闭（`-XX:-UseBiasedLocking`），JDK 18 彻底废弃偏向锁。

### 7. volatile 的作用是什么？能保证原子性吗？

**【阿里/腾讯/华为高频】**

**答：**

**两大核心作用：**

1. **保证可见性**：对 volatile 变量的写操作会立即刷新到主内存，读操作会从主内存重新读取，不使用 CPU 缓存。

2. **禁止指令重排序**：通过插入**内存屏障**（Memory Barrier）实现：
   - volatile 写前插入 `StoreStore` 屏障
   - volatile 写后插入 `StoreLoad` 屏障（全能屏障，开销最大）
   - volatile 读后插入 `LoadLoad` + `LoadStore` 屏障

**不能保证原子性：**
`i++` 是复合操作（读取→加1→写入），volatile 无法保证其原子性。应使用 `AtomicInteger.getAndIncrement()` 或 `synchronized`。

**经典应用 - DCL 单例（双重检验锁）：**
```java
public class Singleton {
    private volatile static Singleton instance;
    
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton(); // 分3步：分配内存→初始化→赋值引用
                }
            }
        }
        return instance;
    }
}
```
volatile 防止步骤2、3被重排序（即防止"赋值了但未初始化"的中间状态被其他线程读到）。

### 8. synchronized 和 ReentrantLock 的区别？

**【阿里P6+/字节高频】**

**答：**

| 对比维度 | synchronized | ReentrantLock |
|---------|-------------|---------------|
| 实现层面 | JVM 关键字 | JDK API（java.util.concurrent） |
| 锁获取方式 | 自动获取和释放 | 手动 lock()/unlock()，需 finally 释放 |
| 等待可中断 | 不支持 | 支持 `lockInterruptibly()` |
| 公平锁 | 非公平 | 支持公平锁（`new ReentrantLock(true)`） |
| 条件变量 | 一个隐式条件（wait/notify） | 多个 Condition（`newCondition()`） |
| 锁状态查询 | 不支持 | `isLocked()`、`getQueueLength()` |
| 超时获取 | 不支持 | `tryLock(long timeout, TimeUnit)` |

**选型建议：**
- 简单同步场景优先用 `synchronized`（JDK 6+ 已优化，语义更清晰）
- 需要高级特性（可中断、超时、多条件）时用 `ReentrantLock`

### 9. 线程池的核心参数和工作原理是什么？

**【阿里/腾讯/美团/京东必考题】**

**答：**

**ThreadPoolExecutor 七大核心参数：**
```java
new ThreadPoolExecutor(
    int corePoolSize,        // 核心线程数（长期保留）
    int maximumPoolSize,     // 最大线程数
    long keepAliveTime,      // 空闲线程存活时间
    TimeUnit unit,           // 时间单位
    BlockingQueue workQueue, // 任务队列
    ThreadFactory threadFactory,    // 线程工厂
    RejectedExecutionHandler handler // 拒绝策略
)
```

**工作流程：**
1. 任务来了，若当前线程数 < corePoolSize → 创建核心线程执行
2. 核心线程满了 → 任务放入 workQueue 排队
3. 队列也满了，且当前线程数 < maximumPoolSize → 创建非核心线程执行
4. 全满了 → 触发**拒绝策略**

**四种拒绝策略：**
| 策略 | 行为 |
|------|------|
| AbortPolicy（默认） | 抛出 RejectedExecutionException |
| CallerRunsPolicy | 由调用者线程执行该任务（降速，不丢任务）|
| DiscardPolicy | 直接丢弃任务，不抛异常 |
| DiscardOldestPolicy | 丢弃队列中最老的任务，重新提交新任务 |

**常用阻塞队列：**
- `LinkedBlockingQueue`：无界队列（谨慎！OOM 风险）
- `ArrayBlockingQueue`：有界队列（推荐生产）
- `SynchronousQueue`：不存储任务，直接交给线程（CachedThreadPool 用）

**线程池大小如何设置？**
- CPU 密集型：`核心数 + 1`（+1 是为了防止偶尔的线程挂起）
- IO 密集型：`核心数 × 2` 或 `核心数 / (1 - 阻塞系数)` （阻塞系数0.8~0.9）

> **大厂实践**：禁止使用 `Executors.newFixedThreadPool()`（无界队列），必须用 `ThreadPoolExecutor` 显式指定参数，并监控队列积压情况。

### 10. 什么是 AQS？ReentrantLock 如何基于 AQS 实现？

**【腾讯/字节/华为架构必考】**

**答：**

**AQS（AbstractQueuedSynchronizer）** 是 JUC 中构建锁和同步器的核心框架，基于它实现了 ReentrantLock、CountDownLatch、Semaphore、CyclicBarrier 等。

**核心原理：**
- 用一个 `volatile int state` 表示同步状态（0=未锁，≥1=已锁/计数）
- 维护一个 CLH（Craig, Landin, Hagersten）**FIFO 双向等待队列**
- 提供 `acquire()`（获取）/ `release()`（释放）等模板方法
- 子类通过实现 `tryAcquire()`/`tryRelease()` 定义具体逻辑

**ReentrantLock 实现（非公平锁）流程：**
1. `lock()` 调用 `tryAcquire()`，用 CAS 将 state 从 0 改为 1
2. 成功 → 设置当前线程为独占线程
3. 失败 → 将线程封装为 Node 加入等待队列，调用 `LockSupport.park()` 挂起
4. `unlock()` 调用 `tryRelease()`，将 state 减 1，为 0 时唤醒队首节点

**可重入实现：** 同一线程再次加锁时 state+1，解锁时 state-1，为 0 才真正释放。

### 11. ThreadLocal 的原理和内存泄漏问题

**【阿里/腾讯/字节高频】**

**答：**

**原理：**
- 每个线程都有一个 `ThreadLocalMap`（类似于 HashMap，弱引用 key）
- `ThreadLocal` 的 `get()/set()` 操作的是当前线程的 `ThreadLocalMap`
- key 是 ThreadLocal 实例本身（弱引用），value 是存储的值

**内存泄漏原因：**
- `ThreadLocalMap` 中 key（ThreadLocal）是弱引用，在没有强引用时会被 GC 回收，导致 key=null 的 Entry 残留（僵尸 Entry）
- 若线程是线程池中的线程（不会被销毁），这些 value 就永远无法被回收

**解决方案：** 每次使用完必须调用 `ThreadLocal.remove()`

```java
ThreadLocal<User> userThreadLocal = new ThreadLocal<>();
try {
    userThreadLocal.set(user);
    // ... 业务逻辑
} finally {
    userThreadLocal.remove(); // 必须！
}
```

**应用场景：** Spring 的 `@Transactional` 通过 ThreadLocal 存储数据库连接；MyBatis SqlSession 管理；MDC 日志追踪。

## 三、Spring 框架

### 12. Spring IoC 和 DI 的原理是什么？

**【阿里/腾讯/华为/百度高频基础题】**

**答：**

**IoC（控制反转）**：将对象的创建和依赖管理权从应用程序代码转移给 Spring 容器，实现解耦。

**DI（依赖注入）**：IoC 的实现方式，容器在创建对象时主动将其依赖的对象注入。

**注入方式：**
- **构造器注入**（推荐）：依赖不可变，线程安全，可以防止循环依赖
- **Setter 注入**：可选依赖
- **字段注入（@Autowired）**：代码简洁，但不利于测试

**Bean 的生命周期（核心12步）：**
1. 实例化（`createBeanInstance`）
2. 属性填充（依赖注入，`populateBean`）
3. 执行 `BeanNameAware`/`BeanFactoryAware`/`ApplicationContextAware` 回调
4. `BeanPostProcessor.postProcessBeforeInitialization()`（前置处理）
5. 执行 `@PostConstruct` 方法
6. 执行 `InitializingBean.afterPropertiesSet()`
7. 执行自定义 `init-method`
8. `BeanPostProcessor.postProcessAfterInitialization()`（后置处理，**AOP 代理在此创建**）
9. Bean 使用阶段
10. 执行 `@PreDestroy` 方法
11. 执行 `DisposableBean.destroy()`
12. 执行自定义 `destroy-method`

### 13. Spring AOP 的实现原理是什么？

**【阿里/腾讯高频】**

**答：**

AOP（面向切面编程）通过**动态代理**实现，在不修改原始代码的情况下横切关注点（日志、事务、权限等）。

**两种代理方式：**

| 类型 | 条件 | 原理 | 性能 |
|------|------|------|------|
| JDK 动态代理 | 目标类实现了接口 | 基于 `java.lang.reflect.Proxy`，运行时生成代理类 | 较高 |
| CGLIB 代理 | 目标类无接口（或配置强制CGLIB） | 字节码增强，生成目标类的子类 | 稍低（首次较慢，运行后相当） |

> Spring Boot 2.0 起默认使用 CGLIB 代理（即使有接口）

**关键概念：**
- **Aspect（切面）**：切点 + 通知的模块
- **Pointcut（切点）**：定义在哪些方法上织入（表达式匹配）
- **Advice（通知）**：Before/After/AfterReturning/AfterThrowing/Around（环绕最强）
- **JoinPoint（连接点）**：可以被拦截的方法执行点

**AOP 失效场景（大厂常考坑点）：**
1. 同一个类内部方法调用（`this.method()`，不经过代理）
2. `@Transactional` 加在 private/final 方法上
3. 多线程中调用（新线程拿不到代理对象）

### 14. Spring 事务传播机制有几种？最常见的坑是什么？

**【阿里/美团/字节架构面试真题】**

**答：**

**7种传播行为：**

| 传播类型 | 含义 | 适用场景 |
|---------|------|---------|
| `REQUIRED`（默认） | 有事务就加入，没有就新建 | 大多数场景 |
| `REQUIRES_NEW` | 始终新建事务，挂起当前事务 | 独立审计日志（不受主事务回滚影响）|
| `SUPPORTS` | 有事务就加入，没有就非事务执行 | 只读查询 |
| `NOT_SUPPORTED` | 非事务执行，挂起当前事务 | 纯查询，不影响事务 |
| `MANDATORY` | 必须在事务中执行，否则抛异常 | 强制要求调用方有事务 |
| `NEVER` | 不能在事务中执行，否则抛异常 | 禁止事务 |
| `NESTED` | 创建嵌套事务（保存点），子事务回滚不影响外层 | 批量操作局部回滚 |

**@Transactional 常见失效场景（大厂必考）：**

1. **方法非 public**：Spring AOP 只代理 public 方法
2. **同类内部调用**：`this.save()` 走的是原始对象，不经过代理
3. **异常被 catch 吞掉**：事务感知不到异常，不回滚
4. **rollbackFor 设置错误**：默认只回滚 `RuntimeException` 和 `Error`，`checked` 异常不回滚
5. **数据库引擎不支持事务**：MyISAM 无事务
6. **多线程**：子线程不在父线程的事务中

```java
// 正确写法：回滚所有异常
@Transactional(rollbackFor = Exception.class)
public void save() throws Exception { ... }
```

### 16. Spring 如何解决循环依赖？

**【阿里/腾讯/字节高频考点】**

**答：**

Spring 通过**三级缓存**解决单例 Bean 的 Setter/Field 注入循环依赖：

| 缓存 | Map 名称 | 存放内容 |
|------|---------|---------|
| 一级缓存 | `singletonObjects` | 完全初始化好的 Bean |
| 二级缓存 | `earlySingletonObjects` | 提前暴露的半成品 Bean（已实例化，未完成属性填充）|
| 三级缓存 | `singletonFactories` | Bean 工厂，用于生成代理对象 |

**流程：** A 依赖 B，B 依赖 A：
1. 创建 A → 实例化 A → 将 A 的工厂放入三级缓存
2. 注入 A 的属性，发现需要 B → 创建 B
3. 实例化 B → 注入 B 的属性，发现需要 A → 从三级缓存取出 A 的工厂，生成 A 的引用（可能是代理）放入二级缓存
4. B 完成初始化 → 放入一级缓存
5. A 获得 B 的引用，完成初始化 → 放入一级缓存

**不能解决的循环依赖：**
- 构造器注入循环依赖（实例化阶段就需要依赖，还没到暴露阶段）
- prototype 作用域循环依赖

### 18. MySQL 索引原理，B+ 树为什么比 B 树更适合数据库？

**【阿里/腾讯/华为/美团必考】**

**答：**

**B+ 树 vs B 树的优势：**

1. **IO 效率更高**：B+ 树非叶节点只存索引键不存数据，扇出更大，树更矮（千万行数据高度约3-4层），查找只需 3-4 次磁盘 IO
2. **查询性能稳定**：B+ 树所有查询都必须走到叶子节点，查询路径长度固定；B 树可能在非叶节点命中，性能不稳定
3. **范围查询高效**：B+ 树叶子节点通过双向链表连接，范围查询只需找到起始节点后顺序遍历；B 树范围查询需要反复回溯
4. **顺序 IO 友好**：叶子节点有序，全表扫描就是顺序读链表

**为什么不用哈希？**
- 不支持范围查询（> < BETWEEN）
- 不支持排序（ORDER BY）
- 不支持联合索引前缀匹配

**聚簇索引 vs 非聚簇索引：**
- **聚簇索引**（主键索引）：叶子节点存储完整的行数据，每张表只有一个
- **非聚簇索引**（二级索引）：叶子节点存储主键值，查询时需**回表**到聚簇索引获取完整数据

### 19. 什么是覆盖索引、索引下推？如何优化慢查询？

**【阿里/字节/美团高频实战题】**

**答：**

**覆盖索引（Covering Index）：**
查询所需的所有字段都在索引中，无需回表。`EXPLAIN` 中 `Extra` 列显示 `Using index`。

```sql
-- 联合索引 (name, age)，以下查询覆盖索引，无需回表
SELECT name, age FROM user WHERE name = '张三';
```

**索引下推（Index Condition Pushdown，ICP）：**
MySQL 5.6 引入。在存储引擎层过滤索引中能判断的条件，减少回表次数。`EXPLAIN` 中 `Extra` 显示 `Using index condition`。

**慢查询优化步骤（大厂实战）：**

1. **开启慢查询日志**：`slow_query_log=1`，`long_query_time=1`
2. **EXPLAIN 分析执行计划**，重点看：
   - `type`：`const > eq_ref > ref > range > index > ALL`（ALL=全表扫描，需优化）
   - `key`：实际使用的索引
   - `rows`：预估扫描行数
   - `Extra`：`Using filesort`/`Using temporary` 是危险信号
3. **常见优化手段：**
   - 针对 WHERE/JOIN/ORDER BY 字段建立索引
   - 使用覆盖索引，避免 SELECT *
   - 避免在索引列上使用函数/运算（导致索引失效）
   - 小表驱动大表，使用 `STRAIGHT_JOIN` 控制驱动顺序
   - 大数据量分页用**延迟关联**：`SELECT * FROM t INNER JOIN (SELECT id FROM t WHERE ... LIMIT 100000, 10) AS tmp USING(id)`

**索引失效场景（大厂必考）：**
- 最左前缀原则被违反
- 对索引列做函数运算：`WHERE YEAR(create_time) = 2024`
- 隐式类型转换：字段是 varchar，条件是数字
- `LIKE '%xxx'`（前导模糊匹配）
- `OR` 连接非索引列
- `NOT IN`、`!=`、`<>` 在低选择度情况下

### 20. MySQL 事务的 MVCC 原理是什么？

**【阿里/腾讯/字节架构面试核心考点】**

**答：**

**MVCC（Multi-Version Concurrency Control）多版本并发控制：**

InnoDB 通过 MVCC 实现不加锁的读（快照读），大幅提升并发性能。

**核心组件：**

1. **隐藏字段**：每行记录有 `DB_TRX_ID`（最近修改的事务ID）、`DB_ROLL_PTR`（回滚指针，指向 undo log）、`DB_ROW_ID`（隐藏主键）

2. **undo log**：记录数据修改前的版本，形成**版本链**

3. **Read View（读视图）**：每次一致性读时创建，包含：
   - `m_ids`：当前活跃（未提交）的事务ID列表
   - `min_trx_id`：活跃事务中最小ID
   - `max_trx_id`：下一个将分配的事务ID
   - `creator_trx_id`：创建该 Read View 的事务ID

**可见性规则：** 读取某行版本时，根据 DB_TRX_ID 判断：
- `DB_TRX_ID < min_trx_id`：该版本在 Read View 创建前已提交，**可见**
- `DB_TRX_ID >= max_trx_id`：该版本在 Read View 创建后才开启，**不可见**
- `min_trx_id <= DB_TRX_ID < max_trx_id`：若在 m_ids 中则**不可见**（未提交），否则**可见**

**不同隔离级别：**
- `READ COMMITTED`：每次快照读都生成新的 Read View → 能读到其他事务已提交的数据
- `REPEATABLE READ`（默认）：首次快照读生成 Read View，后续复用 → 同一事务内读取一致

> InnoDB RR 级别通过 MVCC + Next-Key Lock 很大程度上解决了幻读问题。

### 21. MySQL 如何进行分库分表？ShardingSphere 如何实现？

**【阿里/腾讯/京东大厂架构必考】**

**答：**

**什么时候需要分库分表：**
- 单表数据量 > 1000万行，查询明显变慢
- 单库并发连接数接近瓶颈（MySQL 默认 max_connections=151）
- 磁盘容量不足

**分片策略：**

| 策略 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| 范围分片 | id < 1000w 到分片1 | 范围查询方便 | 数据热点问题 |
| 哈希分片 | id % 分片数 | 数据均匀 | 范围查询困难，扩容麻烦 |
| 一致性哈希 | 虚拟节点环 | 扩容影响小 | 实现复杂 |
| 按地区/时间 | 业务维度划分 | 业务隔离好 | 跨分片查询难 |

**分库分表带来的挑战：**
1. **跨分片查询**：聚合、排序、分页变复杂，需要在中间件层做合并
2. **分布式事务**：跨库操作需要分布式事务（2PC/TCC/Saga）
3. **全局 ID**：不能再用自增 ID，需要雪花算法等
4. **数据迁移**：扩容时需要重新分片

**ShardingSphere 核心原理：**
- SQL 解析 → 分片路由（根据分片键定位到具体库表）→ SQL 改写（修改表名）→ 执行 → 结果合并
- 支持垂直分片（按功能）和水平分片（按数据量）

## 五、Redis 缓存

### 22. Redis 为什么这么快？单线程为何能支撑高并发？

**【阿里/腾讯/字节/百度高频】**

**答：**

**Redis 快的核心原因：**
1. **纯内存操作**：数据在内存，读写纳秒级；磁盘数据库毫秒级，相差千倍
2. **I/O 多路复用 + 单线程事件循环**：基于 Reactor 模式，一个线程通过 epoll 同时监控大量连接的 I/O 事件，避免多线程的上下文切换和加锁开销
3. **高效数据结构**：SDS（简单动态字符串）、ziplist、quicklist、skiplist 等，根据数据大小自动选择编码
4. **RESP 协议简洁**：序列化/反序列化开销小

**单线程不会成为瓶颈吗？**
Redis 的瓶颈不在 CPU，而在内存带宽和网络 IO。单线程避免了锁竞争，代码逻辑简单，CPU 利用率极高。

**Redis 6.0 引入多线程：**
针对网络数据的读写引入多线程（IO Thread），但命令的执行仍是单线程。这样既发挥了多核处理网络 IO 的能力，又保持了命令执行的线程安全。

### 23. Redis 缓存穿透、缓存击穿、缓存雪崩如何解决？

**【阿里/腾讯/字节/美团/京东必考题】**

**答：**

**缓存穿透（查询不存在的数据）：**
- 现象：大量请求查询数据库无记录，缓存无法拦截，直接打到 DB
- 解决：
  1. **布隆过滤器**（推荐）：预先把合法 ID 加入布隆过滤器，非法请求直接拒绝
  2. **缓存空值**：缓存 null，设置短过期时间（5分钟），有数据时更新

**缓存击穿（热点 key 过期）：**
- 现象：热点 key 失效瞬间，大量并发请求同时打到 DB
- 解决：
  1. **互斥锁**：缓存失效时只让一个线程重建缓存（`SETNX`），其他线程等待或返回旧数据
  2. **逻辑过期**（推荐大流量场景）：不设置 TTL，在 value 中存过期时间，异步更新

**缓存雪崩（大量 key 同时过期）：**
- 现象：大量缓存在同一时间失效，DB 被打垮
- 解决：
  1. **TTL 随机化**：`expire = baseTime + random(0, 300)`
  2. **Redis 高可用**：哨兵/集群，防止 Redis 宕机
  3. **本地缓存**（Caffeine/Guava）+ 分布式缓存多级缓存架构
  4. **熔断降级**：服务降级返回兜底数据

### 24. Redis 持久化 RDB 和 AOF 的区别？如何选择？

**【阿里/华为/百度高频】**

**答：**

| 特性 | RDB（快照） | AOF（追加日志） |
|------|-----------|----------------|
| 触发方式 | `SAVE`/`BGSAVE`/配置触发 | 每次写命令后追加 |
| 文件大小 | 小（二进制压缩） | 大（文本指令） |
| 恢复速度 | 快 | 慢（需重放命令）|
| 数据安全性 | 低（两次快照之间数据会丢）| 高（最多丢 1s 数据，取决于 fsync 策略）|
| 文件可读性 | 不可读 | 可读（Redis 命令）|

**AOF 重写（AOFREWRITE）：** 将 AOF 文件中多条命令合并为最终结果的一条命令，减小文件体积。

**混合持久化（Redis 4.0+）：**
AOF 重写时，将当前 RDB 数据写入 AOF 文件头，后续新增命令以 AOF 格式追加。兼顾恢复速度和数据安全性。**生产推荐开启。**

**生产选型：**
- 数据安全要求高（如缓存金融数据）→ AOF（`appendfsync everysec`）
- 允许分钟级数据丢失 → RDB
- **最佳实践**：开启混合持久化

### 25. 如何基于 Redis 实现分布式锁？Redisson 原理？

**【阿里/腾讯/字节高频实战】**

**答：**

**简单实现（SETNX + 过期时间）：**
```
SET lock_key unique_value NX PX 30000
```
- `NX`：只在 key 不存在时设置（互斥）
- `PX 30000`：30秒后自动释放（防止死锁）
- `unique_value`：UUID，释放时验证是否是自己的锁

**存在的问题：**
1. 锁超时自动释放，但业务未执行完（需要自动续期）
2. 单点 Redis 宕机，锁丢失

**Redisson 分布式锁原理：**
1. 使用 Lua 脚本（原子性）实现加锁，value 存储线程标识
2. **WatchDog 看门狗**：后台线程每 10s 检查一次，若锁持有者线程仍存活，将过期时间续期至 30s，解决锁超时问题
3. **可重入**：同一线程多次加锁，用 Hash 结构记录重入次数，释放时计数-1为0才删除

**RedLock（多节点场景）：**
向 N个独立 Redis 节点申请锁，超过 N/2+1 个节点成功才认为加锁成功，解决单点问题。（存在争议，Redisson 实现了 RedLock，但 Martin Kleppmann 认为存在时钟漂移问题）

## 六、分布式系统

### 26. CAP 理论和 BASE 理论是什么？

**【阿里/腾讯/华为架构面试必考】**

**答：**

**CAP 理论：** 分布式系统中，以下三者最多只能同时满足两个：

| 特性 | 含义 |
|------|------|
| **C（Consistency）一致性** | 所有节点在同一时刻看到相同的数据 |
| **A（Availability）可用性** | 每个请求都能收到响应（不保证最新数据）|
| **P（Partition Tolerance）分区容错性** | 网络分区时系统仍能运行 |

> 分区容错（P）是分布式系统必须保证的，因此实际是 **CP 和 AP 的选择**：
> - **CP**：Zookeeper（强一致性，选主期间不可用）、HBase
> - **AP**：Eureka（可用性优先，可能读到旧数据）、Cassandra

**BASE 理论（对 CAP 中 AP 的延伸）：**
- **Basically Available（基本可用）**：允许响应时间加长或部分功能降级
- **Soft State（软状态）**：允许系统中存在中间状态，不同节点的数据副本允许不一致
- **Eventually Consistent（最终一致性）**：经过一段时间后，系统数据最终会达到一致

**应用场景：** 电商下单（最终一致）、余额扣减（强一致）

### 27. 分布式事务有哪些解决方案？

**【阿里/腾讯/京东/美团高频架构题】**

**答：**

**1. 2PC（两阶段提交）**
- 准备阶段：协调者询问所有参与者是否可以提交
- 提交阶段：全部同意则提交，有一个拒绝则全部回滚
- **缺点**：同步阻塞、单点故障（协调者）、脑裂问题

**2. 3PC（三阶段提交）**
- 在 2PC 基础上增加 CanCommit 阶段，引入超时机制
- **缺点**：仍存在数据不一致风险，网络开销大

**3. TCC（Try-Confirm-Cancel）【大厂最常用】**
- **Try**：预留资源（冻结库存）
- **Confirm**：确认执行（扣减库存）
- **Cancel**：回滚（释放冻结库存）
- **优点**：性能好，可控性强
- **缺点**：业务侵入性强，需实现三个方法，需要幂等

**4. Saga 模式**
- 将长事务拆分为一系列本地事务，每个本地事务都有对应的补偿事务
- 顺序执行，失败时反向执行补偿事务
- **适合**：业务流程长（如跨系统的业务流程）

**5. 本地消息表（最终一致性）**
- 业务操作和写消息表在同一本地事务
- 消息中间件异步投递，消费端幂等处理
- **简单可靠**，是阿里早期实践方案

**6. MQ 事务消息（RocketMQ）【字节/阿里常用】**
- 发送半消息（prepare）→ 执行本地事务 → 提交/回滚半消息
- Broker 定时回查本地事务状态
- 保证消息发送与本地事务的原子性

**7. AT 模式（Seata）【现在大厂主流】**
- 自动生成反向 SQL 实现回滚，业务代码无侵入
- 原理：执行 SQL 前后记录 undo_log，回滚时执行反向 SQL

### 28. 如何实现接口幂等性？

**【阿里/腾讯/字节高频实战题】**

**答：**

**幂等性：** 一次请求和多次请求产生的结果相同（防止重复提交）

**常见解决方案：**

1. **Token 机制（推荐）**
   - 请求前先获取 token（UUID）存入 Redis
   - 提交时携带 token，服务端用 Lua 脚本原子性地检查并删除 token
   - 若 token 存在则处理，不存在则拒绝（重复请求）

2. **数据库唯一索引**
   - 对幂等字段建唯一索引，重复插入时抛 DuplicateKeyException

3. **乐观锁（版本号）**
   - `UPDATE ... SET status=1 WHERE id=? AND status=0`
   - 只有首次请求能更新成功

4. **Redis SETNX**
   - 以请求唯一标识（如订单号+操作类型）为 key，SETNX 保证只处理一次

5. **业务状态机**
   - 订单状态只允许单向流转，重复请求检查状态不满足则拒绝

### 29. 分布式 ID 如何生成？雪花算法的原理是什么？

**【阿里/腾讯/字节高频】**

**答：**

**常见方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 数据库自增 | 简单 | 单点、性能瓶颈 |
| UUID | 无序，性能好 | 长度长，无序不适合索引 |
| Redis INCR | 性能好，有序 | 依赖 Redis 可用性 |
| **雪花算法**（推荐）| 高性能，趋势递增，可分布式 | 时钟回拨问题 |
| 美团 Leaf | 雪花算法变种，解决时钟问题 | 依赖 Zookeeper |
| 百度 UidGenerator | 雪花算法变种 | 同上 |

**雪花算法（Snowflake）原理：**
生成 64 位 long 整数：

```
| 1位符号位 | 41位时间戳（ms）| 10位机器ID | 12位序列号 |
```
- **41位时间戳**：约可用 69 年（2^41 / (365×24×3600×1000)）
- **10位机器ID**：支持 1024 个节点（5位数据中心 + 5位机器）
- **12位序列号**：同一毫秒内支持 4096 个 ID

**时钟回拨问题：** 机器时间倒退时会生成重复 ID。解决方案：
1. 检测到回拨时等待时钟追上
2. 使用逻辑时钟，利用序列号段补偿
3. 美团 Leaf 使用 NTP 服务保证时钟同步

### 31. 什么是服务熔断和服务降级？有什么区别？

**【阿里/腾讯/字节架构必考】**

**答：**

**服务熔断（Circuit Breaker）：**
- 当下游服务故障达到阈值时，熔断器**自动开启**，后续请求直接拒绝（不再转发到下游），防止故障扩散
- 类比：电路保险丝，过载自动断路
- 状态：关闭（正常）→ 打开（熔断，直接拒绝）→ 半打开（放行少量请求试探）→ 关闭

**服务降级（Fallback）：**
- 当服务不可用或超时时，返回**兜底数据**（默认值、缓存数据、友好提示）
- 主动保护核心服务，牺牲非核心功能

**区别与联系：**
| 维度 | 熔断 | 降级 |
|------|------|------|
| 触发位置 | 调用方感知下游异常后触发 | 调用方主动处理异常 |
| 目的 | 保护下游，防止雪崩 | 保证用户体验，返回兜底 |
| 是否有fallback | 可以有 | 必有（降级逻辑）|
| 关系 | 熔断触发降级 | 降级是熔断的执行结果之一 |

**Sentinel 熔断规则（阿里开源，大厂标配）：**
- 慢调用比例：RT > 阈值且比例超过设定比例
- 异常比例：异常请求数/总请求数 > 阈值
- 异常数：固定时间窗口内异常数超过阈值

### 34. 如何保证消息不丢失？如何保证幂等消费？

**【阿里/腾讯/京东高频实战题】**

**答：**

**消息不丢失（三个阶段）：**

**1. 生产者阶段：**
- RocketMQ：使用**同步发送**，等待 Broker 确认；或开启**事务消息**
- Kafka：设置 `acks=all`（所有副本确认），`retries=3`

**2. Broker 存储阶段：**
- RocketMQ：开启**同步刷盘**（`flushDiskType=SYNC_FLUSH`），数据写入磁盘后才返回
- 主从复制：设置**同步复制**（`brokerRole=SYNC_MASTER`）
- Kafka：设置 `replication.factor≥2`，`min.insync.replicas=2`

**3. 消费者阶段：**
- 业务处理完成后再手动提交 offset（Kafka）/发送 ACK（RocketMQ）
- 禁止自动提交（`enable.auto.commit=false`）
- 消费失败时不要提交，让消息重新投递

**幂等消费（防止重复消费）：**

消费者可能因网络超时重复收到同一条消息，需要保证幂等性：

1. **唯一索引**：消息 ID 存数据库唯一索引，重复消费直接忽略
2. **Redis SETNX**：以消息 ID 为 key，成功才处理
3. **状态机**：检查业务数据状态，已处理则跳过
4. **数据库乐观锁**：`UPDATE ... WHERE id=? AND status=0`

```java
// RocketMQ 消费示例（先查后处理保证幂等）
@RocketMQMessageListener(topic = "order-topic", consumerGroup = "order-consumer")
public class OrderConsumer implements RocketMQListener<OrderMessage> {
    @Override
    public void onMessage(OrderMessage message) {
        // 幂等检查
        if (orderService.isProcessed(message.getMsgId())) {
            return;
        }
        // 业务处理
        orderService.process(message);
    }
}
```

### 35. RocketMQ 事务消息原理是什么？

**【阿里/字节/美团高频考点】**

**答：**

RocketMQ 事务消息保证本地事务和消息发送的原子性（要么都成功，要么都失败）。

**流程（两阶段提交 + 补偿机制）：**

```
1. 发送"半消息"（Half Message）到 Broker（消费者不可见）
2. 执行本地事务（如扣款）
3. 根据本地事务结果发送 Commit 或 Rollback 给 Broker
   - Commit：Broker 将半消息变为正常消息，消费者可消费
   - Rollback：Broker 删除半消息
4. 如果步骤3因网络问题丢失：Broker 定时（默认1分钟）回查本地事务状态
   → 实现 TransactionListener.checkLocalTransaction() 方法
```

**核心组件：**
- `TransactionMQProducer`：事务消息生产者
- `TransactionListener`：
  - `executeLocalTransaction()`：执行本地事务
  - `checkLocalTransaction()`：Broker 回查时执行

**适用场景：** 跨服务的消息和本地事务原子性，如下单后扣减库存。

### 37. 如何设计一个短链系统？

**【字节/腾讯/百度高频设计题】**

**答：**

**需求分析：**
- 长 URL → 短 URL（如 `https://t.cn/AbCdEf`）
- 访问短 URL 302 重定向到原始 URL
- 高读（QPS 万级）、低写

**核心算法（ID 转短码）：**
```java
// 将全局自增 ID 转为 62 进制（0-9,a-z,A-Z）
private static final String CHARS = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";

public String encode(long id) {
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.insert(0, CHARS.charAt((int)(id % 62)));
        id /= 62;
    }
    return sb.toString(); // 6位可支持 62^6 ≈ 570亿个短链
}
```

**存储设计：**
- 数据库：`short_code(varchar PK) → long_url, create_time, expire_time`
- Redis 缓存：`short_code → long_url`，LRU 淘汰

**架构设计：**
```
写：客户端 → 短链服务 → 生成 short_code（雪花ID转62进制）→ DB + Redis
读：客户端 → 短链服务 → Redis 查询 → 302 重定向
                            ↓ miss
                          DB 查询 → 回填 Redis
```

**防滥用：**
- 生成速率限制（IP 维度）
- 同一 long_url 复用同一 short_code（Hash 去重）
- 设置有效期

### 39. 限流算法有哪些？生产中如何实现？

**【阿里/腾讯/字节/百度高频考点】**

**答：**

**四种限流算法：**

**1. 固定窗口（计数器）：**
- 每个时间窗口内记录请求数，超过阈值拒绝
- 缺点：临界问题（窗口切换时可能双倍流量）

**2. 滑动窗口：**
- 将时间窗口细分为多个小格，动态移动窗口
- 解决固定窗口的临界问题，Redis + Sorted Set 实现

**3. 漏桶算法（Leaky Bucket）：**
- 请求入桶，桶以固定速率出水（处理请求）
- 超出桶容量则溢出（拒绝）
- **优点**：严格平滑输出速率
- **缺点**：无法应对突发流量（桶满即拒）

**4. 令牌桶算法（Token Bucket）【推荐】：**
- 令牌以固定速率放入桶，请求消耗令牌
- 桶满时令牌溢出（允许积累一定的突发能力）
- **优点**：允许一定程度的突发
- Guava `RateLimiter`、Sentinel、Nginx 都使用令牌桶

**Redis + Lua 实现滑动窗口限流：**
```lua
local key = KEYS[1]
local window = tonumber(ARGV[1])  -- 窗口大小（毫秒）
local limit = tonumber(ARGV[2])   -- 限制次数
local now = tonumber(ARGV[3])     -- 当前时间戳（毫秒）

-- 清除窗口外的数据
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
-- 统计窗口内请求数
local count = redis.call('ZCARD', key)
if count < limit then
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, math.ceil(window/1000))
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

### 42. 手写 LRU 缓存（大厂高频手写题）

**【阿里/腾讯/字节/百度现场手写高频】**

**答：**

**LRU（Least Recently Used）** 最近最少使用，淘汰最久未使用的数据。

**实现1：使用 LinkedHashMap（5行实现）**
```java
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;
    
    public LRUCache(int capacity) {
        // accessOrder=true：按访问顺序排序（最近访问的在末尾）
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }
    
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;  // 超过容量时移除最老的
    }
}
```

**实现2：HashMap + 双向链表（面试官更喜欢的写法）**
```java
public class LRUCache {
    class Node {
        int key, val;
        Node prev, next;
        Node(int k, int v) { key = k; val = v; }
    }
    
    private Map<Integer, Node> map = new HashMap<>();
    private Node head = new Node(0, 0); // 虚拟头节点
    private Node tail = new Node(0, 0); // 虚拟尾节点
    private int capacity;
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }
    
    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        moveToTail(node);  // 访问后移到尾部（最近使用）
        return node.val;
    }
    
    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.val = value;
            moveToTail(node);
        } else {
            if (map.size() == capacity) {
                // 移除头部（最久未使用）
                Node evict = head.next;
                removeNode(evict);
                map.remove(evict.key);
            }
            Node newNode = new Node(key, value);
            addToTail(newNode);
            map.put(key, newNode);
        }
    }
    
    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }
    
    private void addToTail(Node node) {
        node.prev = tail.prev;
        node.next = tail;
        tail.prev.next = node;
        tail.prev = node;
    }
    
    private void moveToTail(Node node) {
        removeNode(node);
        addToTail(node);
    }
}
```

### 43. 如何在海量数据中找 Top K？

**【阿里/腾讯/百度/华为高频算法题】**

**答：**

**方案1：大根堆（内存可以装下所有数据）**
- 建立大小为 K 的小根堆，遍历所有数据
- 若当前元素 > 堆顶，则替换堆顶并调整
- 时间复杂度：O(n log k)

```java
public int[] topKFrequent(int[] nums, int k) {
    // 统计频率
    Map<Integer, Integer> freq = new HashMap<>();
    for (int n : nums) freq.merge(n, 1, Integer::sum);
    
    // 小根堆（按频率）
    PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[1] - b[1]);
    for (Map.Entry<Integer, Integer> e : freq.entrySet()) {
        pq.offer(new int[]{e.getKey(), e.getValue()});
        if (pq.size() > k) pq.poll(); // 保持堆大小为k
    }
    
    int[] result = new int[k];
    for (int i = k - 1; i >= 0; i--) result[i] = pq.poll()[0];
    return result;
}
```

**方案2：分治 + 快速选择（找第 K 大元素）**
- 利用 QuickSort 的 partition，平均 O(n) 找到第 K 大

**方案3：海量数据（内存不够）**
1. 将数据按 Hash 分片到多台机器
2. 每台机器本地求 Top K（用小根堆）
3. 汇总所有机器的 Top K，再求全局 Top K

### 45. 说说你对 DDD（领域驱动设计）的理解

**【阿里/腾讯 P8+/架构师必考】**

**答：**

DDD 是一种面向复杂业务的软件设计方法论，核心思想是**业务驱动技术，技术服务业务**。

**核心概念：**
- **领域（Domain）**：业务范围，如电商领域
- **限界上下文（Bounded Context）**：明确的业务边界，如订单上下文、库存上下文
- **实体（Entity）**：有唯一标识的对象（如订单 Order）
- **值对象（Value Object）**：无标识、不可变（如地址 Address、金额 Money）
- **聚合根（Aggregate Root）**：聚合的入口，保证业务一致性（如 Order 是订单聚合根）
- **领域服务（Domain Service）**：跨聚合的业务逻辑
- **领域事件（Domain Event）**：聚合内发生的业务事件（如 OrderCreatedEvent）
- **仓储（Repository）**：聚合的持久化接口

**与微服务的关系：**
- 限界上下文 = 微服务边界（一个限界上下文对应一个微服务）
- 领域事件 = 服务间通信（通过 MQ 发布）

**大厂实践：** 阿里、腾讯等大厂在核心业务系统中广泛采用 DDD，将复杂业务拆分为清晰的领域模型，提高代码可维护性。

### 46. 如何做技术方案设计？如何权衡技术选型？

**【阿里P7/腾讯T3-2/架构师面试综合题】**

**答：**

**技术方案设计的关键维度：**

1. **需求分析**：功能需求 + 非功能需求（QPS、延迟、可用性、数据量）
2. **核心流程设计**：画出核心业务流程图和系统交互图
3. **存储设计**：选择合适的数据库（OLTP/OLAP/NoSQL），设计表结构和索引
4. **接口设计**：RESTful / RPC 接口定义
5. **容量估算**：DAU × 操作次数 = QPS，QPS × 数据量 = 存储
6. **高可用设计**：单点故障分析，主从/集群/熔断/降级
7. **性能优化**：缓存层、异步化、读写分离

**技术选型核心原则（STAR）：**
- **S（Scalability）可扩展性**：能否水平扩展
- **T（Trade-off）权衡**：性能 vs 一致性；简单 vs 功能
- **A（Availability）可用性**：SLA 要求（几个9）
- **R（Reliability）可靠性**：数据不丢、异常恢复

**经典问题：** "Redis 还是 DB 存 Session？"
- 分析：多实例需要共享 Session → 集中存储 → Redis（高性能 + 支持过期）
- 权衡：增加了 Redis 依赖，需要保证 Redis 可用性
